# CUSUM/Shewhart トラック 使い方・運用ガイド

`state_logic_cusum.py` を月次で回し、要確認インボックスと Tableau 用テーブルを出すための運用手順。
設計の詳細は `state_logic_cusum_ドキュメント.md`、操作点決めは `backtest_手順.md` を参照。

---

## 1. ファイル構成
同じフォルダに置く（社外に出さない＝ローカル完結）。

| ファイル | 役割 |
|---|---|
| `cusum_monitor.py` | 検知エンジン（ドリフト＝CUSUM、スパイク＝条件付き二項） |
| `earlylife_baseline.py` | 安定化前の期待カーブ lambda0(t) 推定 |
| `state_logic_cusum.py` | 状態判定・リセット注入・インボックス／Tableau 生成（**これを実行する**） |
| `backtest_cusum.py` | 操作点決めのバックテスト（初期設定時に使う） |
| `台帳テンプレート_CUSUM版.xlsx` | 台帳マスタの雛形 |

依存は numpy / pandas のみ。

## 2. 入力データ

### 2.1 パネル（監視データ）
縦長形式。1行＝機種×部番×販社×年月。CSV か Excel。

| 列（既定名） | 内容 |
|---|---|
| 事業コード | 事業区分 |
| 開発コード | 機種 |
| 部番 | 部品番号 |
| 販社 | 販売会社（販社合算で集約される） |
| 年月 | YYYYMM |
| 経過月 | 発売からの経過月（0始まり） |
| 月次使用数 | その月の修理部品出庫数 |
| 累積販売台数 | その時点の母数（fleet） |
| SF-コード | （任意）安定化前の集団キー。部品グループ分類 |
| ランク | （任意）価格帯。集団キーの細分 |

列名が違う場合は CONFIG の `cols` でマッピングする（実データの列名に変更しない）。
SF-コードが無くても安定期は動く。安定化前は SF-コードがあると先行機種カーブを借りられる。

### 2.2 台帳（人間の判断のマスタ）
`台帳テンプレート_CUSUM版.xlsx` をコピーして使う。**追記型・1ファイルがマスタ**。`台帳` シートに1行＝1判断。

| 列 | 必須 | 内容 |
|---|---|---|
| 事業コード／開発コード／部番 | ○ | 対象の監視単位 |
| 判定年月 | ○ | この判断を適用する月（YYYYMM） |
| 記録日 | | 記入日（同月複数判断の順序づけ） |
| 処置区分 | ○ | 8択（→ 5節） |
| 再評価年月 | | 指定月にインボックスへ自動再掲 |
| 上書きR／上書きh | | 「感度上書き」のとき単位個別のR/h |
| 新ベースライン値 | | 「新常態受容」の新lambda0（直接指定） |
| ベースライン窓起点／窓長 | | 「新常態受容」を窓から再推定する場合 |
| 原因メモ／確認者 | | 記録用（処理には不使用） |

機種全体を止める場合は、部番に `機種全体` と入れるか処置区分「機種終了」を使う。

## 3. 初期設定（CONFIG）
`state_logic_cusum.CONFIG` をコピーして編集する。

```python
import state_logic_cusum as s
cfg = dict(s.CONFIG)
cfg["panel_path"]  = "panel.xlsx"        # または .csv
cfg["ledger_path"] = "台帳.xlsx"
cfg["ledger_sheet"] = "台帳"
# 列名が既定と違う場合だけ cols を調整:
# cfg["cols"]["monthly_use"] = "出庫数"   など
# 操作点（バックテストで決めた値を入れる）:
cfg["R"] = 2.0; cfg["h"] = 5.0
cfg["alpha_spike"] = 0.005; cfg["min_count"] = 3; cfg["burst_window"] = 0
```

主なつまみ（詳細は設計ドキュメント2節）：
- `R, h`：ドリフト感度。h が誤報・検出遅れの主つまみ。
- `alpha_spike, min_count, burst_window`：スパイク感度。
- `stable_start_m, baseline_len`：安定期ベースライン窓（経過月）。
- `monitor_end_m`：これを超える末期は監視対象外。
- `min_leaders`：安定化前で必要な先行機種数。未満は監視保留。
- `earlylife_group_levels`：安定化前の集団階層（既定 `[["biz","sf","rank"],["biz","sf"]]`）。

## 4. 月次の回し方

```python
inbox, tableau, meta = s.run(cfg)
# 出力ファイル: cfg["out_inbox"], cfg["out_tableau"]（既定で *_cusum.csv）
```

`run` は最新月（パネルの最大年月）を基準月にする。過去月で再現したいときは `cfg["asof_ym"]=YYYYMM`。

月次サイクル：
1. 最新パネルを用意して `run(cfg)`。
2. `要確認インボックス_cusum.csv` を開いてレビュー。
3. 判断した単位を**台帳に追記**（処置区分・判定年月など）。
4. 翌月、また `run`。台帳が効いて状態が更新される。

状態は保存していない。毎回パネル＋台帳から再生する（だから過去再現できる＝監査に強い）。

## 5. 処置区分メニュー（インボックスにどう記録するか）
インボックスで発火した単位に、原因を見て1つ選んで台帳に書く。

| 処置区分 | いつ使う | 振る舞い | 次の状態 |
|---|---|---|---|
| **対策中** | 真の異常。対策を打った | リセット（判定月の翌月からS=0）。ベースライン維持 | 対策効果確認中。効かず再点灯なら要確認に戻る |
| **新常態受容** | レートが恒久的に変わった（仕様変更等） | リセット＋新ベースライン採用 | 正常 |
| **ノイズ** | 一過性で問題なし | リセットのみ | 監視中 |
| **感度上書き** | この単位だけ閾値を変えたい | リセット＋単位個別の R/h | 監視中 |
| **スパイク確認** | スパイクを確認・問題なし（リセット不要） | 何もしない（再掲だけ抑制） | 現状維持 |
| **保留** | 判断保留・継続監視 | 何もしない | 要確認保留（毎月インボックスに継続表示） |
| **監視終了** | この単位の監視を終える | 単位をクローズ | 終了(単位) |
| **機種終了** | 機種全体を終える | 機種の全部番をクローズ | 終了(機種) |

「対策中」に**再評価年月**を入れておくと、その月に自動でインボックスへ再掲（効果の振り返り）。

## 6. 出力の読み方

### 6.1 要確認インボックス（`要確認インボックス_cusum.csv`）
基準月に対応が要る単位だけ、注目度の降順。台帳に貼り戻しやすい列順。

| 列 | 意味 |
|---|---|
| アラート種別 | drift / spike / burst の組み合わせ |
| 基準モード | 安定期／安定化前／安定化前(暫定)／安定化前(保留) |
| 載る理由 | 発火／保留継続／再評価 |
| S, h | CUSUM 値としきい値 |
| 月次p値 | スパイク検定のp値 |
| 観測使用数／期待故障数 | 当月実績と期待 |
| 注目度 | `max(S/h, log p / log alpha)`。優先度ソートに使う |
| 現ベースライン起点月／リセット回数 | 履歴の手がかり |
| 処置区分〜記録日（空欄） | ここに記入して台帳へ貼る |

### 6.2 Tableau 用テーブル（`tableau_監視テーブル_cusum.csv`）
全単位×全月の時系列。ダッシュボードの入力。主な列：CUSUM・しきい値h・期待故障数・各アラート・総合アラート・
リセット点・判定点・状態・注目度・最新状態・単位注目度・基準モード・先行機種数・集団レベル。

## 7. 状態の意味
- **正常**：発火なし、処置なし。
- **監視中**：処置（ノイズ等）後、発火していない。
- **要確認**：当月発火。対応が要る。
- **要確認保留**：保留中（毎月インボックス継続）。
- **対策効果確認中**：対策中の後。Sが寝たままなら効いた、再点灯なら要確認に戻る。
- **終了(単位)／終了(機種)**：クローズ済み。

## 8. モード（物差しの違い）
- **安定期**：自機種の過去（自己ベースライン）が物差し。経過月が `stable_start_m+baseline_len-1` 以上で成立。
- **安定化前**：自己履歴が足りない若い機種。同じ集団（SF-コード〔＋ランク〕）の先行機種から借りた時変カーブ lambda0(t) が物差し。
- **安定化前(保留)**：先行機種が `min_leaders` 未満で物差しが作れない。鳴らさず保留。先行機種が増えれば自動で監視に入る。
- **安定化前(暫定)**：SF-コード列が無くカーブが作れない。プール平均で暫定。

## 9. よくあるつまずき
- **保留が多い**：SF-コード集団が薄い（先行機種<min_leaders）。`min_leaders` を見直すか、集団を広く取る（ランクを外す＝既に自動フォールバック）。
- **立ち上げ直後に発火が多い**：正常。ノイズ・新常態を台帳で処理すれば翌月から落ち着く。
- **対策したのに翌月また要確認**：データ上のレート上昇が続いている＝対策が効いていないサイン（CUSUM トラックの狙いどおりの挙動）。
- **過去の判定を再現したい**：`cfg["asof_ym"]` に過去のYYYYMMを入れて `run`。台帳もその月までのものだけ効く。
- **列名エラー**：`cols` マッピングを実データの列名に合わせる。
