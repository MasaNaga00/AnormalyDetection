## A. 操作点の確定（最優先）
検知パラメータを実データで決めて CONFIG に固定する。詳細は `backtest_手順.md`。

- [ ] **月間アラート予算を決める**。チームが毎月triageできる新規アラート件数。1件の調査工数（原因調査・販社確認・台帳記録）×担当人数から逆算。ドリフトとスパイクに配分する。
- [ ] **バックテストを実データで回す**。`backtest_cusum.run_backtest("panel.xlsx", cfg)`。`backtest_drift.csv` / `backtest_spike.csv` が出る。
- [ ] （あれば）**既知異常ラベル**（`事業コード,開発コード,部番,発生年月`）を `labels_path` で渡し、実測の検出遅れも確認する。
- [ ] **トレードオフ表から操作点を選ぶ**。`recommend_drift / recommend_spike` に予算を渡して候補を絞る。
  - ドリフト: 捕まえたい最小のレート上昇倍率を決め、その倍率で検出率が十分・月間誤報件数が予算内になる最大の `h` を選ぶ。`R` は最速で捕まえたい倍率に合わせる。
  - スパイク: `alpha_spike`、`min_count`（低カウント部品の偽陽性ガード）、`burst_window`（2〜3月の急増を拾うか）。
- [ ] **CONFIG に反映**。`R, h, alpha_spike, min_count, burst_window`（必要なら `alpha_burst`）。
- [ ] 安定化前カーブのつまみ（`earlylife_smooth_window, earlylife_rate_floor_frac`）は既定で十分なら触らない。気になれば同じ枠組みで感度確認。

## B. 初回本番運用の立ち上げ
詳細は `使い方_運用ガイド.md`。

- [ ] **パネルデータを用意**。縦長形式（`事業コード,開発コード,部番,販社,年月,経過月,月次使用数,累積販売台数`、任意で `SF-コード,ランク`）。社外に出さない＝ローカル完結。
- [ ] **台帳を用意**。`台帳テンプレート_CUSUM版.xlsx` をコピーして使う。初回は空（ヘッダのみ）でよい。
- [ ] **CONFIG のパス・列名を設定**。`panel_path, ledger_path`、`cols` のマッピングを実データの列名に合わせる。
- [ ] **初回実行**。`run(cfg)` → `要確認インボックス_cusum.csv` と `tableau_監視テーブル_cusum.csv`。
- [ ] **初回インボックスをレビュー**。発火している単位を確認し、各々に処置区分を判断（対策中／ノイズ／保留 等）。立ち上げ時はノイズ・新常態が多めに出るのが普通。
- [ ] **モード分類を確認**。安定期／安定化前／安定化前(保留) の振り分けが妥当か。保留が多い場合は先行機種が足りていない（SF-コード集団が薄い）サイン。

## C. 月次運用の定着
- [ ] **月次サイクルを確立**：最新パネル取得 → `run(cfg)` → インボックスをレビュー → 台帳に処置を追記 → 翌月へ。
- [ ] **処置の記録ルールを共有**。8つの処置区分の使い分け（`使い方_運用ガイド.md` の処置区分メニュー）。台帳は追記型・1ファイルマスタ。
- [ ] **再計算で過去判定の再現を確認**（任意）。`asof_ym` を過去月にして `run` すると当時のインボックスが再現される（監査・振り返り用）。
- [ ] **対策効果の確認運用**。「対策中」にした単位が、リセット後にSが寝たままか（効いた）／再点灯するか（効かず＝再度要確認）を翌月以降に見る。CUSUM トラック固有の価値。

## D. Tableau ダッシュボード構築（CUSUM版）
固定しきい値トラックで作ったチェックリストの CUSUM 版。入力は `tableau_監視テーブル_cusum.csv`。

- [ ] **ワークリスト**：最新状態が要確認／要確認保留／対策効果確認中の単位を、単位注目度の降順で。
- [ ] **単位ドリルダウン**：機種×部番を選ぶと CUSUM 系列・期待故障数・月次使用数の推移、リセット点・判定点のマーカー。
- [ ] **ヒートマップ**：機種×部番 × 年月 を注目度で色分け。ドリフトの広がりを俯瞰。
- [ ] **診断ビュー**：基準モード・先行機種数・集団レベルで、安定化前単位がどの物差しで見られているかを確認。
- [ ] **状態の凡例・色**：正常／監視中／要確認／要確認保留／対策効果確認中／終了。

## E. 精緻化・拡張（運用が回り始めてから）
- [ ] **先行機種から異常期間を除外**。現状は全安定期単位を leaders にしている。台帳で「対策中／監視終了」になった期間を leaders から除くと、初期不良カーブの上振れを防げる。
- [ ] **ランクの段階導入**。`earlylife_group_levels` の階層フォールバックが既に対応済み。ランクを付けた集団は自動的にランク単位に、薄い／欠損はSF単位に落ちる。Tableau の「先行機種数／集団レベル」で厚みを見ながら拡げる。
- [ ] **厳密な監査再現が要るなら**：安定化前カーブを asof 時点でスナップショット固定する拡張（先行機種が若い集団でのみ意味がある。`state_logic_cusum_ドキュメント.md` 限界参照）。
- [ ] **再評価年月の活用**。「対策中」に再評価年月を入れておくと、その月にインボックスへ自動再掲（載る理由＝再評価）。
