# state_logic_cusum.py ドキュメント — CUSUM/Shewhart 異常検知の状態判定運用

部品出庫データ（datamart_A）と台帳（Excel）を入力に、CUSUM（ドリフト）＋ Shewhart（スパイク）
の異常検知について各監視単位の状態を判定し、「要確認インボックス」と「Tableau用監視テーブル」を
出力するモジュール。固定しきい値トラックの `state_logic.py` に対応する CUSUM 版で、設計原則
（台帳をマスタにした判断の記録、監視単位＝機種×部番への集約、機種ごと対象外など）はそのまま踏襲する。

本書はその実装仕様にあたる。検知の数式そのもの（CUSUM 再帰・スパイク検定）は
`cusum_monitor.py` に、安定化前の時変カーブ `lambda0(t)` は `earlylife_baseline.py` に委譲する
（フェーズ1・フェーズ2とも結線済み）。本モジュールはその上に
**リセット注入・ベースライン差し替え・状態解決・インボックス／Tableau 生成**を載せる層である（7章「接続点」参照）。

---

## 1. 何をするものか（全体像）

毎月の運用サイクルの中で、このモジュールが担うのは「機械判定」の部分だけ。

```
datamart_A ──┐
             ├─→ state_logic_cusum.py ──→ 要確認インボックス_cusum.csv ──→ 人がレビューし台帳に追記
台帳.xlsx ───┘            │
                          └────────────→ tableau_監視テーブル_cusum.csv ──→ Tableauで可視化
```

- 判断（原因の調査、処置区分の選択、再アラート条件の決定）は人が台帳に書く。
- 本モジュールは台帳を読んで「各監視単位の現在の有効な条件」を機械的に解決し、
  CUSUM を**リセット・ベースライン差し替えを織り込んでリプレイ**し、鳴っているかどうかを決めるだけ。
- 状態（正常／監視中／対策効果確認中／要確認／終了）はどこにも保存しない。毎回、台帳＋パネルから
  計算で導く。→ 台帳が唯一のマスタであり、台帳を直せば過去の状態も再計算で正しくなる。

### 固定しきい値トラックとの対応

| 概念 | 固定しきい値（state_logic.py） | CUSUM/Shewhart（本モジュール） |
|---|---|---|
| 監視指標 | 累積使用率(%) | CUSUM統計量 `S_t`（ドリフト）／月次p値（スパイク） |
| トリガ | 率 ≥ 当月閾値 | `S_t ≥ h` ／ `p_t ≤ alpha_spike ∧ count ≥ min_count` |
| 鳴り止ませる機構 | 閾値を Y に上げる | **CUSUM を 0 にリセット**（スパイクは当月記録の冪等性で抑制） |
| 新常態の受容 | 該当なし（率は下がらない） | **ベースライン `lambda0` を再推定**（人間の明示判断） |
| 改善の確認 | 不可（累積率は下がらない） | 可（リセット後に `S` が寝たままか観察）← 本トラックの強み |
| マスタ／過去再現 | 台帳＋`asof_ym` | 同左 |

## 2. 入力

### 2.1 パネル（datamart_A のエクスポート）

縦長の月次パネル。1行 = 事業コード × 開発コード × 部番 × 販社 × 年月。
列名は `CONFIG["cols"]` でマッピングする。

| 内部名 | 既定の列名 | 役割 |
|---|---|---|
| biz | 事業コード | キー |
| dev | 開発コード | キー（機種） |
| part | 部番 | キー |
| dist | 販社 | キー |
| ym | 年月 | YYYYMM（整数/文字列/日付様式いずれも可。内部でYYYYMM整数に正規化） |
| elapsed | 経過月 | 発売からの経過。ベースライン窓・モード分類の時間軸 |
| monthly_use | 月次使用数 | 監視する生カウント |
| cum_sales | 累積販売台数 | 期待故障数 mu0 の分母（保有台数近似） |

### 2.2 台帳（Excel）

`台帳テンプレート_CUSUM版.xlsx` の「台帳」シート。追記専用ログで、1行 = 1回の確認イベント。
読み込み時に判定年月・再評価年月・ベースライン窓起点を YYYYMM 整数へ、各数値パラメータを数値へ正規化する。

本モジュールが**判定に使う**列は次のとおり：
事業コード / 開発コード / 部番 / 判定年月 / 記録日（同月内の順序づけ）/ **処置区分** /
再評価年月 / 上書きR / 上書きh / 新ベースライン値 / ベースライン窓起点 / ベースライン窓長。

その他の列（アラート種別・基準モード・S・h・月次p値・観測使用数・期待故障数・経過月・注目度・
現ベースライン起点月・リセット回数・原因メモ・確認者）は、インボックスから転記する**記録用**で、
判定には使わない。判定に効くのは「処置区分」と、それに紐づくパラメータだけ。

## 3. 処理の流れと、各段階でやっている評価

### 3.1 前処理: 月次0補完（`_fill_zero_months`）

各系列（事業×開発×部番×販社）の初出月〜最終月の抜けを 0 行で埋める安全網。
月次使用数=0、累積販売台数・経過月は前方補完。DBから1系列1月1行で取れている限り発火しない。

### 3.2 監視単位の集約（`aggregate_units`）

主監視単位＝**機種×部番（販社合算）**。使用数・累積販売台数とも販社合計を取り、
販社の偏りに左右されない感度優先の系列にする（販社別の副監視は二段目の課題）。

### 3.3 ベースライン推定とモード分類（`estimate_baseline` / `classify_mode`）

- **安定期モード**: 自己ベースライン。ベースライン窓（経過月 `[stable_start_m, +baseline_len-1]`）の
  使用数合計 ÷ 販売台数合計で `lambda0`（1台あたり月次故障ペース）を推定し**固定**する。
  移動平均更新はドリフトを吸ってしまうので採らない。
- **安定化前モード**: 最大経過月が `stable_start_m + baseline_len - 1` 未満の若い機種。
  先行機種（同集団の**安定期**単位）から時変カーブ `lambda0(t)` を推定して借りる
  （`earlylife_baseline.py`、フェーズ2で結線済み）。`build_earlylife_curves` が安定期単位だけを
  leaders として集団ごとに1本のカーブを作り、`earlylife_lambda0` が監視対象の集団値で引き当てる。
  - **集団の階層フォールバック**（`earlylife_group_levels`、細→粗）: 既定は `[["biz","sf","rank"],["biz","sf"]]`。
    監視対象ごとに、キーが揃い（ランク欠損でない）かつ先行機種が `min_leaders` 以上ある**最初の階層**を採用する。
    ランクで十分な先行機種がいればランク単位、ランクが薄い／欠損ならSF単位に**自動で落ちる**。
    → ランクが一部の機種にしか無くても動く。ランクを付けた集団だけ自動的に細かくなる（採用階層は Tableau の「集団レベル」列に出る）。
  - ランクが全く無いデータ（rank列が無い）でも、ランク階層は自動でスキップされSF単位で動く。
  - leaders は安定期単位のみ＝監視対象自身は入らないので、**スケールは先行機種側で固定**される
    （自身の初期データでスケールを合わせ直さない＝異常をカーブが吸わない）。
  - 先行機種が `min_leaders` 未満の集団は**監視保留**（カーブ全0＝監視不能）。安定期レートの平均で
    代用すると初期高を無視して誤報を量産するので、代用はしない。基準モード列に「安定化前(保留)」と出す。
  - 集団キー列（SF-コード）が無いパネルでは従来の暫定（プール平均スカラ）にフォールバックし
    「安定化前(暫定)」と明示する。

### 3.4 検知エンジン（`cusum_monitor.py` に委譲・フェーズ1結線済み）

ドリフトとスパイクの計算は `cusum_monitor.py` に委譲する。本モジュールはセグメント単位で呼び出す。

- **ドリフト（`cusum_monitor.poisson_cusum`）**: `k_t = (R-1)·lambda0_t·fleet_t / ln R`、
  `S_t = max(0, S_{t-1} + count_t - k_t)`、アラート `S ≥ h`。保有台数増は k に吸収され、レート上昇分だけに反応。
  `lambda0` はスカラ（安定期）でも配列カーブ（安定化前）でも受ける。
  **`reset_after_alarm=False` で呼ぶ**：本体の既定（発火月に自動でS=0へ戻す）は鋸歯状になり未対応ドリフトを
  谷間で見落とす。自動リセットは切り、リセットは台帳由来（reset-after-M）だけにする。台帳のリセット点で
  タイムラインをセグメントに分割し、各セグメントを S=0 から呼び直すことで reset-after-M を再現する。
- **スパイク（`cusum_monitor.spike_test`）**: 安定期はベースライン窓の `(C, E)=(使用数合計, 販売台数合計)` を
  渡し、**条件付き二項検定**（2つのポアソンレート比較の正確検定）で判定する。短いベースラインによる
  lambda0 過小推定での誤報を、検定自体が保守側に吸収する（素朴ポアソン裾より決して鳴りやすくならず、
  ベースラインが薄いほど保守側に開く）。安定化前は配列 lambda0 を渡すと自動で各月の直接ポアソン検定に切り替わる。
  `p ≤ alpha_spike ∧ count ≥ min_count` で点灯。
- **バースト（同 `spike_test` の `burst_window≥2`）**: 直近 W 月の合算カウントを同方式で検定。
  単月では足りずCUSUMには短すぎる2〜3月の急増を拾う中間レンジ。各セグメント先頭 W-1 月は窓が
  揃わず NaN（リセット直後はバースト窓も再開）。

### 3.5 台帳イベント → リプレイ計画（`build_plan` / `ReplayPlan`）— リセット注入の心臓部

監視単位の台帳イベントを時系列に解決し、リプレイ計画を組む。処置区分ごとの振る舞いは
`DISPOSITIONS` 表に集約されている（4章）。計画が持つもの：

- `reset_after`: **reset-after-M**。判定年月 M の当月 S は発火として残し、M の翌月への
  繰り越しを 0 にする（監査ログを残すため当月は消さない）。
- `baseline_from`: ベースライン再推定の発効点 `[(翌月, 新lambda0, C, E)]`。新ベースライン値が直接
  指定されればそれ（C,E は現窓を流用）、無ければ窓（起点＋窓長）から `lambda0・C・E` を推定、
  それも無ければ現状維持。C,E は新セグメントのスパイク条件付き二項に渡る。
- `override_from`: 感度上書きの発効点 `[(翌月, R, h)]`。この単位だけ R/h を差し替える。
- `close_month`: 監視終了（単位）の発効月。

リセット・ベースライン差し替え・感度上書きはいずれも**判定年月の翌月から有効**にし、reset-after と
整合させている。

### 3.6 CUSUM リプレイ（`replay_unit`）

監視レンジ（安定期 `[stable_start_m, monitor_end_m]`、安定化前 `[0, monitor_end_m]`）に絞り、
台帳のリセット点で**セグメントに分割**する（reset-after-M：M はセグメント末尾に含め、翌月から新セグメント）。
好都合なことに、リセットを伴う処置区分はすべてセグメント境界になり、`lambda0/C/E` や `R/h` を変える処置
（新常態受容・感度上書き）は同時にリセットも起こすので、**1セグメント内ではこれらが一定**。各セグメントは
有効な `(lambda0, C, E)`（`baseline_at`）と `R/h`（`override_at`）を引き、`S=0` から
`cusum_monitor.poisson_cusum`（`reset_after_alarm=False`）と `spike_test` を呼んで結果を連結する。
CUSUM は再帰統計だが、起点からのリプレイに台帳の発効点を注入するだけなので、状態を保存しない／
`asof_ym` で過去再現できる、という性質が保たれる（`unit_events` は `判定年月 ≤ asof` のイベントだけ honor する）。

返す月次量: `S / mu0(期待故障数) / k / lambda0 / 月次p値 / バーストp値 / ドリフトアラート /
スパイクアラート / バーストアラート / 総合アラート / リセット点 / 基準モード`。

### 3.7 状態解決（`resolve_state`） — 判定の心臓部

任意の監視単位と月 t に対し「状態・判定種別・再評価到来・当月発火」を返す。優先順位：

```
1. 機種終了が発効済み(発効月 ≤ t)            → 終了(機種)。判定しない
1'. 監視終了が発効済み(発効月 ≤ t)           → 終了(単位)。判定しない
2. 台帳に t 以前の行が無い                    → 正常（未記録）。鳴れば「初回」
3. t 以前の最新行の処置区分は何か?
   ├ 保留        → 要確認保留（毎月インボックス）
   ├ 対策中      → 対策効果確認中
   └ ノイズ/新常態受容/感度上書き → 監視中
4. 当月 S≥h もしくはスパイク点灯              → 状態を「要確認」に引き上げ（鳴っている月）
5. 最新行の再評価年月 ≤ t なら「再評価到来」フラグ
```

「対策効果確認中」は固定しきい値トラックには無い状態で、対策後に CUSUM が寝たままかを
観察する＝改善を確認できるという本トラックの価値そのもの。

### 3.8 要確認インボックス（`build_inbox`）

基準月 asof で**人がレビューすべき単位だけ**を一覧化。載る条件：

| 理由 | 条件 |
|---|---|
| 発火（初回/再） | 当月にドリフト or スパイクが点灯、かつ当月未記録 |
| 保留継続 | 最新行が「保留」（鳴っているかに関係なく毎月載る） |
| 再評価 | 最新行の再評価年月 ≤ asof（S が動かなくても時間で載せる） |

載らない: 終了(単位/機種)、当月すでに記録済みで保留でない（冪等性）。

各行に付く情報: アラート種別 / 基準モード / 判定種別（初回|再）/ 載る理由 / S / h / 月次p値 /
観測使用数 / 期待故障数 / 経過月 / **注目度** / 現ベースライン起点月 / リセット回数、および空の
人間記入列（処置区分以降）。**注目度** = `max(S/h, log p / log alpha_spike)`。どちらも 1.0 超で
発火を意味し、ドリフトとスパイクを同一スケールで順位付けできる。worklist はこの降順でソート。

このCSVの列は台帳の自動転記列と同じ並びなので、レビュー後はこの行を台帳にコピーし、
処置区分以降の人間記入列を埋めれば1件の記録が完了する。

### 3.9 Tableau用監視テーブル（`build_tableau`）

単位×月の全行に注釈を付けた縦長テーブル。主要列：

| 列 | 意味 |
|---|---|
| 月次使用数 / 累積販売台数 | 監視する生カウントと分母 |
| lambda0 / 期待故障数 / k | その月時点で有効な平常レート・mu0・基準増分 |
| CUSUM / しきい値h / R | CUSUM 軌跡とその時点の感度。台帳の記入に従い階段状に変わる |
| 月次p値 | スパイク検定の上側p値 |
| ドリフト/スパイク/総合アラート | 各エンジンと総合の点灯 |
| リセット点 | その月にリセットを注入したか（リセットが効いた月のマーカー） |
| 判定点 | その月に台帳の記録があるか（人が確認した月のマーカー） |
| 状態 | 終了 / 要確認 / 対策効果確認中 / 監視中 / 正常（その月時点） |
| 最新状態 | **単位レベル**: 最新月時点の状態。ヒートマップの行フィルタはこちらを使う |
| 注目度 / 単位注目度 | 各月の注目度と、最新月の注目度（行ソート用） |
| 基準モード / 現ベースライン起点月 | 何を平常とみなして比較しているか |
| リセット回数 / 直近リセット月 | 慢性的にリセットしていないか（本物のドリフトを消す兆候）の手掛かり |

## 4. 処置区分メニューと再アラート条件

調査結論として人が1つ選ぶ。これと付随パラメータだけが判定に効く。

| 処置区分 | CUSUM | ベースライン | 結果状態 | 再点灯のしかた |
|---|---|---|---|---|
| 対策中 | リセット | 維持（旧lambda0） | 対策効果確認中 | 対策が効けば寝たまま。効かず再び S≥h なら要確認。再評価年月で時間確認も可 |
| 新常態受容 | リセット | 再推定（新lambda0） | 正常（新基準） | 新基準で通常監視。上昇を基準化するので必ず明示判断で |
| ノイズ | リセット | 維持 | 監視中 | 同じ h で通常監視。すぐ再点灯すれば誤報でなかった証拠 |
| 感度上書き | リセット | 維持 | 監視中 | この単位だけ h'/R' で監視。乱用注意・理由必須 |
| スパイク確認 | 操作なし | 維持 | 監視中/正常 | 当月記録の冪等性で再表示が止まる。別月の再発は新事象 |
| 保留 | 操作なし | 維持 | 要確認保留 | 結論が出るまで毎月インボックスに残す |
| 監視終了 | — | — | 終了(単位) | 以降この単位は判定しない |
| 機種終了 | — | — | 終了(機種) | 部番=機種全体 で記録。以降その機種の全単位を対象外 |

- **新常態受容**は「新ベースライン値」を直接書くか、「ベースライン窓起点＋窓長」を書けば窓から
  再推定する（両方空なら現状維持）。自動再推定はしない＝問題のマスクを避けるため必ず人の明示判断。
- **スパイク**は累積しないのでリセットの概念が要らない。高カウントが継続するならそれはドリフトであり
  CUSUM 側が拾う（両エンジン併走の意味）。

## 5. 使い方

### 5.1 デモ実行（実データなし）

```bash
python state_logic_cusum.py
```

組み込みの疑似データ＋台帳記入例で動き、PART-A の CUSUM 軌跡を**リセットあり/なしで並べて**表示する

各部品の最新状態とインボックスも表示する。

### 5.2 実データでの実行

1. `CONFIG["panel_path"]` に datamart_A エクスポートのパスを設定
2. `CONFIG["cols"]` を実データの列名に合わせる
3. `CONFIG["ledger_path"]` に台帳 xlsx のパスを設定
4. `R / h / alpha_spike / min_count` と安定期ベースライン（`stable_start_m / baseline_len`）を
   backtest.py の操作点決定結果に合わせる
5. `python state_logic_cusum.py` → 2つの CSV が出る

### 5.3 毎月の運用手順

```
1. datamart_A の最新エクスポートを所定パスに置く
2. python state_logic_cusum.py
3. 要確認インボックス_cusum.csv を開き、各単位を調査
```

`CONFIG["asof_ym"] = 202603` のように基準月を指定すると、その月時点のインボックスを再現できる。
台帳が追記専用である限り、いつ実行しても同じ結果になる（監査・振り返り用）。

## 6. 仕様上の注意・既知の限界

- **リセットが中心概念**: CUSUM は累積するので、放置すると h を超えたまま鳴り続ける。固定しきい値の
  「Y に上げる」に相当する操作がリセット。リセットは reset-after-M（当月の発火は残す）。
- **慢性的なリセットは要注意**: 短期間に何度もリセットしている単位は、本物のドリフトを誤報判定で
  消している可能性がある。インボックス／Tableau の「リセット回数」で可視化している。
- **ベースライン再推定は人間の明示判断のみ**: 自動再推定は上昇をそのまま基準にして問題をマスクする。
  Tableau に「現ベースライン起点月」を出し、何と比較しているかを常に見えるようにしている。
- **安定化前モードは先行機種カーブに結線済み（フェーズ2）**: 同集団の安定期単位から `lambda0(t)` を
  借りる。先行機種が `min_leaders` 未満なら監視保留（鳴らさない）。集団キー列が無ければプール平均で暫定。
- **スパイク検定は条件付き二項に結線済み（フェーズ1）**: 短いベースラインで lambda0 を過小推定した
  単位での誤報を、検定自体が保守側に吸収する（`cusum_monitor.spike_test`、3.4）。
- **asof再現性の範囲**: 状態は保存せず台帳から再生するので、過去月を基準に再実行すれば当時の判定が
  再現される。安定期は自己ベースラインが系列早期に固定されるので完全再現。安定化前は先行機種から
  借りるカーブを使うため、**先行機種が成熟していれば**若い経過月のカーブは安定し再現も完全だが、
  先行機種自体が若く一緒に育っている場合はデータ増加でカーブが後から精緻化され、過去の lambda0(t) が
  わずかに動きうる（ベースライン再推定と同根）。厳密な監査再現が要る場合は asof 時点のカーブを
  スナップショットして固定する拡張余地がある。
- **妥当性ウィンドウ**: 累積販売台数は末期に過大化し、上側 CUSUM では本物の上昇をマスクする方向に効く。
  経過月 `monitor_end_m` を超える末期は監視対象外（さらに「監視終了」で台帳に記録も可）。
- **同月複数行**: 同じ単位・同じ判定年月に複数行あれば記録日の新しい行が有効。記録日は必ず入れること。
- **部番=機種全体 の行は単位の履歴に混ぜない**: 機種終了の解決にだけ使う。

## 7. cusum_monitor.py / earlylife_baseline.py との接続点（結線状況）

本モジュールは状態判定・リセット注入の層であり、検知の数式は既存エンジンに委譲する。**両エンジンとも結線済み**。

| 結線 | 役割 | 状況 |
|---|---|---|
| ドリフト → `cusum_monitor.poisson_cusum` | 時変平均ポアソン上側CUSUM（`reset_after_alarm=False`、セグメント呼び） | **フェーズ1 結線済み** |
| スパイク/バースト → `cusum_monitor.spike_test` | 安定期=条件付き二項（窓の C,E）／安定化前=直接ポアソン | **フェーズ1 結線済み** |
| 安定化前 → `earlylife_baseline.estimate_earlylife_curve` + `attach_curve_to_unit` | 先行機種群から時変カーブ `lambda0(t)` を推定し配列で `replay_unit` へ | **フェーズ2 結線済み** |

安定化前の結線（フェーズ2）は `build_earlylife_curves`（leaders=安定期単位から集団ごとにカーブ生成）と
`earlylife_lambda0`（監視対象の集団値で引き当て・`min_leaders` 未満は保留）で実装。`spike_test` は
配列 lambda0 を検知して自動で直接ポアソンに切り替わるため、検定側の追加対応は無い。`replay_unit` は
セグメントごとに配列カーブを経過月で切り出して渡す。

### 検証結果（再現可能。`verify_*.py` 同梱）
- **フェーズ1 回帰**: 委譲版ドリフト S と設計式の独立再計算が**完全一致**（最大絶対差 0）。
- **フェーズ1 条件付き二項**: x=4 固定で `C=8` のとき素朴ポアソン p=4.9e-3（α=0.005で誤発火）に対し
  条件付き二項 p=1.05e-2（沈黙）。薄いほど保守側に開く（C=1 で 87倍）。
- **フェーズ2 カーブ結線**: 先行機種3つから推定した `lambda0(t)` が真のカーブを追従。正常な新機種は
  鳴らず、異常な新機種（2.6倍）は**経過月1で点灯**。先行機種1つの集団は**保留**（監視不能で鳴らさない）。
- **フェーズ2 スケール固定**: 異常な監視対象を leaders 母集団から抜き差ししてもカーブの差は **0**
  （監視対象はカーブ推定に一切影響しない）。
- **フェーズ2 ランク階層フォールバック**: ランクが一部の機種にしか無いデータで、ランクA（先行機種3）の
  単位は `biz+sf+rank`、ランクB（先行機種1）とランク欠損の単位は `biz+sf`（先行機種6）に自動で落ちることを確認。
- **フェーズ3 統合**: 安定期5単位＋安定化前2単位を1回の実行で評価し、台帳の対策中/ノイズのリセット注入と
  状態解決がモードに依らず効くことを確認（正常→正常、ドリフト→対策後も継続なら再点灯=要確認、
  ノイズ→監視中、正常な新機種→正常、異常な新機種→要確認）。
- **フェーズ3 asof再現性**: 同一パネルで asof を変えても asof以前の月の S・発火・状態が**完全一致**（差0、
  状態を保存せず台帳から再生）。パネルを asof で打ち切ったリアルタイム再現でも、安定期・安定化前とも
  この設定では差0（先行機種が成熟していれば、若い経過月のカーブは打ち切りに残る古いデータから作られるため）。

### 拡張余地
- leaders に台帳で異常と判定された期間が混じるとカーブが上振れする。実運用では「監視終了/対策中」期間を
  leaders から除外する拡張余地がある（現状は全安定期単位を leaders にする素直版）。
- ランク差が大きい集団は `earlylife_group_levels` の最細階層 `["biz","sf","rank"]` で分離される（既定で有効）。
  ランクが薄い／欠損の集団は自動でSF単位に落ちるので、ランクは段階導入でよい。集団の厚みは Tableau の
  「先行機種数」「集団レベル」列で確認する。
