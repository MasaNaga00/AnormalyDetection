手順は3ステップです。「一軍入り月」はすでに `simulate_capped_triage` が記録していて、インシデント別CSVの**「インボックス入り月」列がそれ**です（発火候補として一軍に載った最初の月。Nに依存しません。Nの影響は「top-N処理月」に出ます）。

**① 入力を2つ用意**

```python
import pandas as pd
import state_logic_cusum as s
from simulate_capped_triage import simulate_capped_triage

cfg = dict(s.CONFIG)
cfg.update(R=1.5, h=5.0, alpha_spike=0.005, min_count=3, burst_window=0)  # 確定した操作点

panel = s._prepare_panel(pd.read_csv("実パネル.csv"), cfg)   # 手順Aの整形済みパネル
units = s.aggregate_units(panel)
labels = pd.read_csv("実ラベル.csv", encoding="utf-8-sig")
# ラベルの列: 事業コード, 開発コード, 部番, 発生年月（=販社から指摘があった月）
```

**② 決めたNで1本回す**

```python
monthly, incidents, details, ledger = simulate_capped_triage(
    units, cfg,
    top_n=10,                      # 決めた月間チェック件数N
    assumed_disposition="対策中",   # 発火→台帳記録の消し込み方の想定
    labels=labels,
    lookback_m=6,                  # 報告月の6ヶ月前から捕捉を探す（backtestと同じ流儀）
    collect_details=True)          # ③のドリルダウン用
incidents.to_csv("インシデント検証.csv", index=False, encoding="utf-8-sig")
```

`incidents` がそのまま答えの表で、ラベル1件＝1行です。

| 列 | 読み方 |
|---|---|
| インボックス入り月 | **一軍入りした月**（発火候補になった最初の月） |
| インボックス遅れ月 | 一軍入り月 − 販社報告月。**負＝販社より早く一軍に載っていた** |
| top-N処理月 / 処理遅れ月 | 実際にチェックが回ってきた月とその遅れ。N=10の影響はここに出る |
| キャップ起因の追加遅れ | 処理月 − 一軍入り月。予算不足で待たされた月数 |
| 検知＝なし | 期間内に一度も一軍に載らなかった＝**R/hの問題**（Nを増やしても直らない） |
| 処理＝なし | 一軍には載ったが期間内に上位Nに入れなかった＝**Nの問題** |

**③ 個別インシデントの経過を月単位で追う**

「なぜこの月まで処理が回らなかったのか」を見たいときは `details` を絞ります。

```python
d = details[(details["開発コード"]=="S04") & (details["部番"]=="P1")]
print(d[["シミュレーション年月","載る理由","注目度","S","観測使用数"]].to_string(index=False))
```

各月に「発火（一軍に載った）／再評価／保留継続」のどれだったかと注目度が出るので、「発火し続けたが注目度○位で3ヶ月押し出されていた」のような経過が読めます。さらに順位まで見たい月は、その月の `monthly` の発火候補数と `details` の同月・発火行を注目度降順に並べれば何位だったか分かります。

**判定の切り分け方**：検証結果が悪かったとき、原因は3層に分かれます。「検知＝なし」→操作点（R/h）を backtest 側で見直す。「インボックス遅れ月が大きい」→これも操作点の感度の話。「キャップ起因の追加遅れが大きい」→Nの話で、ここだけがN増で改善します。この切り分けができるのがこの表の主目的です。

Nをまだ確定していない・比較もしたい場合は、前回の `run_sweep(units, cfg, n_list=[5,10,15,None], labels=labels, ...)` にすれば同じ表がN縦積みで出て、そのままTableauでN別に並べられます。

はい、その2段構えで正しいです。順序の意図も合っています。整理すると、

**第1段（`compare_spike_onoff`, top_n=None）＝「スパイクを消していいか」の判断**

`top_n=None`（無制限）にすることで、キャップの影響を排除して**検知能力そのものだけ**を比較できます。ここで見るのはスパイクオン/オフで「オンのみ検知」が何件出るか、早期性が落ちるものがどれだけあるか。これがクリアなら「スパイクを消す」という**操作点の構成をここで確定**させます。

**第2段（`simulate_capped_triage`, top_n を振る）＝「予算Nをどこに置くか」の判断**

スパイク構成を確定させた後、その確定cfg（オフならalpha_spike=Noneのまま）で `top_n` を振って処理タイミングのトレードオフを見る。この順序が正しいのは、**Nを決めるべき対象の発火母集団が第1段の結論で変わる**からです。スパイクをオフにすれば発火候補が減り、同じNでも積み残しや処理遅れが変わります。先にNを決めてしまうと、スパイク構成を変えたときにNを決め直すことになります。

一点だけ補足しておくと、第1段で「スパイクを残す」と結論した場合は、第2段は `alpha_spike=0.005` のまま `top_n` を振る。「消す」と結論したら `alpha_spike=None` で振る。どちらに転んでも、**第1段で確定したcfgをそのまま第2段に渡す**という受け渡しさえ守れば一貫します。

それと、第2段は前回の `run_sweep` を使えば `top_n` のスイープが一発で縦積みCSVになるので、`simulate_capped_triage` を手でN回呼ぶより楽です。

```python
# 第1段の結論を反映した cfg（例：スパイクオフに決めた場合）
cfg_final = dict(cfg); cfg_final["alpha_spike"] = None
from simulate_capped_triage import run_sweep
run_sweep(units, cfg_final, n_list=[5,10,15,20,None], labels=labels,
          assumed_disposition="対策中")
```

第1段が終わったら結果を持ってきてください。「オンのみ検知」が0件か数件かで第2段の入口（オフで振るかオンで振るか）が決まります。
