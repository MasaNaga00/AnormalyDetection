# 実運用スタートガイド（top_n=15・上位15件triage運用）

急ぎ実運用を始めるための、最小手順に絞った実務ドキュメント。
「実行 → 課内で部品の合否判断 → 台帳に記録 → 翌月また実行」のループを回す。

前提となる確定事項（このガイドはこれを所与とする）:
- 操作点は確定済み（R, h, alpha_spike, min_count, burst_window を CONFIG に反映済み）。
- 予算 N = 15（過去データで「販社に勝てる」ことを確認済みの水準）。
- スパイク検定の扱い（オン/オフ）も CONFIG に反映済み。

---

## 0. 一番大事な対応関係（これだけ先に理解する）

**シミュレータの top_n=15 と、実運用の「上位15件を見る」は同じこと。**

`run()` が出力する `要確認インボックス_cusum.csv` は **注目度の降順にソート済み**。
だから実運用で予算N=15を実現するのは、

  **インボックスの上から15行だけを課内でレビューする**

これだけ。simulate_capped_triage の top_n はあくまで過去データでの検証用であって、
月次運用のプログラムに N を渡す場所は無い。運用側は「出力インボックスの上位N行を見る」
という人間の運用ルールで N を効かせる。

持ち越しも自動で正しく効く。16位以下で今月見送ったものは、本物なら CUSUM の S が
翌月さらに積み上がって注目度が上がり、翌月のインボックスで上位に来る
（reset_after_alarm=False の設計思想どおり）。だから「見送った＝取りこぼし」ではなく
「順番待ち」であって、放置しても本物は勝ち上がってくる。

## 1. 事前準備（初回だけ）

### 1-1. ファイルを同じフォルダに置く
- `state_logic_cusum.py`（本体）
- `cusum_monitor.py` / `earlylife_baseline.py`（検知エンジン）
- 実パネル（機種×部番×販社×月）… CSV か Excel
- 台帳マスタ Excel（`台帳テンプレート_CUSUM版.xlsx` をコピーして使う。初回は「台帳」シートが0行）

### 1-2. CONFIG を確定値にする
`state_logic_cusum.py` の CONFIG を開き、次を実データ用に設定する。

```python
CONFIG = {
    # 入出力
    "panel_path": "実パネル.csv",          # or .xlsx
    "panel_sheet": 0,                       # Excelのときのシート名/番号
    "ledger_path": "台帳_CUSUM.xlsx",       # 台帳マスタのパス（下の1-3で用意）
    "ledger_sheet": "台帳",
    "out_inbox": "要確認インボックス_cusum.csv",
    "out_tableau": "tableau_監視テーブル_cusum.csv",
    # 検知パラメータ（確定済みの操作点を入れる）
    "R": ...,           # 確定値
    "h": ...,           # 確定値
    "alpha_spike": ..., # 確定値（オフに決めたなら None）
    "min_count": ...,
    "burst_window": ...,
    "reset_after_alarm": False,  # ← 変更しない
    "asof_ym": None,             # 最新月を自動採用。過去再現したい時だけ指定
    # 以下は既定のままでよい（ベースライン/レンジ/安定化前）
    ...
}
```

列名が既定と違う場合は CONFIG の `cols`（biz/dev/part/dist/ym/elapsed/monthly_use/
cum_sales/sf/rank）を実データのヘッダに合わせる。

### 1-3. 台帳マスタを用意する
`台帳テンプレート_CUSUM版.xlsx` をコピーして `台帳_CUSUM.xlsx` にリネーム。
「台帳」シートは 0 行（ヘッダのみ）の状態でよい。コードが読む列は以下:

  事業コード / 開発コード / 部番 / 判定年月 / 記録日 / 処置区分 /
  再評価年月 / 上書きR / 上書きh / 新ベースライン値 / ベースライン窓起点 / ベースライン窓長

「選択肢」シートの8処置区分プルダウンはそのまま使える（誤入力防止）。

**注意**: CONFIG の `ledger_path` は必ずこの 0 行台帳ファイルのパスを指すこと。
`ledger_path=None` にすると内蔵のデモ台帳が読み込まれてしまい、実運用にならない。
初回も「0行の台帳Excelを指定」で始める（Noneにしない）。

### 1-4. パネルの投入前チェック（実データ品質）
初回だけ、パネルに次の問題が無いか確認する。
- **年月の書式**: "2023-02" のようなハイフン区切りは自動変換されるが、ゼロ埋めなし
  ("2023-2") が混じると5桁誤変換になる。文字列長のばらつきを確認。
- **累積販売台数の非単調**: 数値YYYYMMでソートし直してから確認。本物の逆転が残る場合は
  機種×部番×販社ごとに cummax() で単調非減少に補正してから使う。

```python
import pandas as pd, state_logic_cusum as s
df = pd.read_csv("実パネル.csv")
df["年月"] = df["年月"].map(s.to_yyyymm)
df = df.sort_values(["事業コード","開発コード","部番","販社","年月"])
df["累積販売台数"] = df.groupby(["事業コード","開発コード","部番","販社"])["累積販売台数"].cummax()
df.to_csv("実パネル.csv", index=False, encoding="utf-8-sig")
```

## 2. 初回実行（今の溜まりを一気に洗い出す特別回）

空台帳（0行の台帳Excel）で1回 `run()` する。これまで見ていなかった分の異常が塊で出るので、
件数が15件を大きく超えても慌てない。上位15件から着手し、残りは翌月以降に持ち越す
（本物は順位が上がって戻ってくる）。

初回も CONFIG の `ledger_path` は 0 行の台帳Excelを指したまま（1-3参照）。

```python
import state_logic_cusum as s
inbox, tableau, meta = s.run(s.CONFIG)
print(f"インボックス総数 {len(inbox)} 件（うち今月見るのは上位15件）")
print(inbox.head(15)[["開発コード","部番","アラート種別","載る理由",
                       "注目度","S","観測使用数","経過月"]].to_string(index=False))
```

`要確認インボックス_cusum.csv` と `tableau_監視テーブル_cusum.csv` が出力される。

## 3. 月次ループ（毎月これを回す）

### ステップ1: 最新パネルを用意
今月のデータを足したパネルを 1-4 のチェックにかけて上書き保存。

### ステップ2: 実行
```python
inbox, tableau, meta = s.run(s.CONFIG)
```

### ステップ3: 上位15件を課内でレビュー（合否判断）
`要確認インボックス_cusum.csv` を開く。**注目度降順で並んでいるので上から15行**が今月の
対象。各行を課内で見て、部品ごとに「これは本物の異常か／ノイズか／対応中か」を判断する。

各行の読み方:
- **載る理由**: 発火（新規/累積で今アラート）/ 再評価（対策中の効果確認期限）/ 保留継続
- **アラート種別**: drift（じわ上がり）/ spike（単月急増）/ drift+spike
- **注目度**: 高いほど優先。S/h とスパイクp値から算出
- **S / h**: CUSUM統計量としきい値。S が h を超えて発火
- **観測使用数 / 期待故障数**: 実際の出庫 vs 期待。乖離が異常の大きさ
- **経過月**: 発売からの月数。小さい＝安定化前（初期不良期の可能性）
- **判定種別**: 初回 / 再（過去に処置済みで再び載った）

16位以下は今月は見送る。放置してよい（本物なら翌月上位に来る）。ただし件数が毎月
15件を大きく超え続け、積み残しが単調増加するなら、それは操作点が発火を出しすぎている
サインなので、落ち着いたら R,h の見直しか issue（再点灯問題）の対処を検討する。

### ステップ4: 台帳に記録
レビューした部品について、**インボックスの該当行をそのまま台帳の「台帳」シートに貼り付け**、
処置区分だけ埋める（プルダウンで選ぶ）。判定年月・記録日はインボックスの値を使う。

処置区分8択と効果:

| 処置区分 | いつ使う | 効果 |
|---|---|---|
| 対策中 | 本物の異常。対策を打つ | リセット。再評価年月に効果確認（効かず再点灯なら要確認） |
| 新常態受容 | 上昇後の水準を正常とみなす | リセット＋新ベースライン。以後その水準で監視 |
| ノイズ | 一過性・誤報と判断 | リセットのみ |
| 感度上書き | この単位だけ R/h を変えたい | リセット＋単位個別 R/h（上書きR・上書きh 列に記入） |
| スパイク確認 | 単月急増を確認済み、対策不要 | リセットなし・再掲抑制（※後述の注意） |
| 保留 | 今は判断できない/様子見 | 毎月インボックスに継続表示 |
| 監視終了 | この部番の監視をやめる | その単位をクローズ |
| 機種終了 | 機種全体を終える | 部番="機種全体" で機種全体クローズ |

**対策中を選んだら「再評価年月」を入れる**（例: 3ヶ月後）。その月に効果確認として
自動でインボックスに戻ってくる。

台帳を上書き保存する。

### ステップ5: Tableau更新（共有する場合）
`tableau_監視テーブル_cusum.csv` を固定パスに置いておけば、Tableau はデータソース更新
（Ctrl+Alt+F5）だけで最新になる。

### 翌月はステップ1に戻る
台帳に記録した処置は次回 `run()` で自動反映される（リセット・再評価・保留継続など）。
状態はプログラムが毎回パネル＋台帳から再生するので、途中経過を保存する必要はない。

## 4. 運用上の注意（先に知っておくと事故らない）

### 4-1. スパイク確認 → ドリフト再点灯（既知の挙動）
大きな単月スパイクは CUSUM(ドリフト)側も S を積み増す。「スパイク確認」はリセットなしなので、
S が高止まりして翌月以降ドリフトとして再点灯し続けることがある。当面の運用対処:
- 単月急増を確認して対策不要なら、「スパイク確認」ではなく **リセットの伴う区分**
  （原因が分かって対処するなら「対策中」、一過性なら「ノイズ」）を選ぶと再点灯しない。
- 「スパイク確認」を使って翌月ドリフトで再点灯したら、その時に改めて処置する。

### 4-2. 早期段階（安定化前）のラベルは順位で沈みやすい
発売から日が浅い部番（経過月が小さい）は、注目度が安定期の派手なドリフトに負けて
15位以内に入りにくいことがある。検知はできている（インボックスには載る）が、上位15件の
外に出やすい。当面は上位15件運用で回しつつ、早期段階の取りこぼしが問題になりそうなら、
「経過月が小さい行だけ別途ざっと目視する」を追加すると拾える。

### 4-3. 見送りは取りこぼしではない
上位15件の外は「今月は順番が来なかった」だけ。本物は S が伸びて翌月上位に来る。
毎月ゼロベースで判断するのではなく、持ち越しは自然に競り上がる仕組みだと理解しておく。

### 4-4. 台帳のバックアップ
台帳は唯一のマスタ。月次で記録したら、日付つきでコピーを残す
（例: `台帳_CUSUM_2025-07.xlsx`）。asof再現性の監査にも効く。

## 5. 困ったときの最小チェック

- **インボックスが空 / デモ機種が混ざる**: パネルの最新月が想定どおりか（asof_ym=None で
  最新月自動）。年月書式の誤変換で監視レンジがずれていないか。台帳に見覚えのない機種が
  混ざるなら ledger_path が None になっていてデモ台帳を読んでいる（0行台帳を指す）。
- **前は載っていた部品が載らない**: 台帳で既にリセット/対策中記録が入っていないか
  （対策中は再評価年月まで沈黙する）。
- **件数が異常に多い**: 初回は正常（溜まりの一括洗い出し）。2回目以降も多いなら
  操作点が緩すぎる可能性。落ち着いてから R,h を再検討。
- **特定ラベルの検知タイミングを確認したい**: simulate_capped_triage や
  analyze_capped_labels 等の検証ツール群を使う（急ぎでなければ）。
