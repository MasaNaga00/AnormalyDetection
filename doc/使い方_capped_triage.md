# simulate_capped_triage 使い方（上位N件triage＋二軍ウォッチの実測）

## 目的
「厳しい閾値のまま、毎月の正規triageを注目度上位N件に絞り、残りは持ち越して再競争。
二軍（ウォッチリスト）はTableau目視のみ」という運用を過去データで月送り再現し、
**N vs 検知遅れのトレードオフ曲線**と「販社に勝てた件数」をNごとに出す。

## なぜ速くなったか（fast版が1時間超だった理由）
fast版は静的前処理は外出ししたが、**replay_unit（CUSUM全履歴再計算）を毎月×全単位で
やり直していた**（2000単位×36ヶ月=72,000回）。replayは台帳イベントが増えた時しか
変わらない（asof再現性で実証済みの性質）ので、増分キャッシュ化した：

- 初期リプレイは全単位1回だけ（台帳空の状態）
- 月ループは ym→行index のO(1)ルックアップ＋当月だけのresolve_state
- 再リプレイは「その月に台帳注入された単位」だけ（＝アラーム件数回）

**実測（合成2000単位×36ヶ月）**：前処理5秒＋初期リプレイ13秒＋月ループ3秒/N。
N=5本のスイープ全体で47秒。進捗は前処理・初期リプレイ・月ループの各段階で
経過秒と残り目安を表示する。

## 実データでの手順

```python
import pandas as pd
import state_logic_cusum as s
from simulate_capped_triage import run_sweep

cfg = dict(s.CONFIG)
cfg.update(R=..., h=..., alpha_spike=..., min_count=..., burst_window=...)  # 確定操作点

panel = s._prepare_panel(pd.read_csv("実パネル.csv"), cfg)   # 手順A整形済みのもの
units = s.aggregate_units(panel)
labels = pd.read_csv("実ラベル.csv", encoding="utf-8-sig")
# 列: 事業コード, 開発コード, 部番, 発生年月（=販社からの報告月）

monthly, incidents = run_sweep(
    units, cfg,
    n_list=[5, 10, 15, 20, 30, None],   # None=無制限（上限側の参照点）
    labels=labels,
    watch_frac=0.5,                # 二軍の線: 注目度>=0.5（S/hの半分 or logスケールでalphaの半分）
    assumed_disposition="対策中",   # 楽観側は "ノイズ" でもう1周
    lookback_m=6)                  # backtestと同じ流儀（報告6ヶ月前から捕捉を探す）
# → capped_月次負荷.csv / capped_インシデント別.csv（N縦積み、そのままTableauへ）
```

## 出力の読み方

### capped_インシデント別.csv（1行=ラベル×N）
| 列 | 意味 |
|---|---|
| ウォッチ入り月 / 遅れ月 | 二軍で目視可能になった最初の月（インボックス入りより前の分だけ） |
| インボックス入り月 / 遅れ月 | 発火候補として一軍に載った最初の月。**Nに依存しない**（キャップは検知を変えない） |
| top-N処理月 / 処理遅れ月 | 実際にtriageが回ってきた月。**Nの影響はここに出る** |
| キャップ起因の追加遅れ | 処理月 − インボックス入り月。予算不足の純粋なコスト |
| 販社より早い | 処理遅れ月<0（報告前に正規triage完了）なら○ |
| 検知 / 処理 | なし=期間内に一度も載らなかった / 一度も上位Nに入らなかった |

遅れ月は「捕捉月 − 販社報告月」。**負＝先回り成功**。

### N別サマリの出し方（Nの決め方）
```python
g = incidents.groupby("月間予算N").agg(
    処理率=("処理", lambda x: (x=="あり").mean()),
    処理遅れ中央値=("処理遅れ月", "median"),
    キャップ追加遅れ中央値=("キャップ起因の追加遅れ", "median"),
    販社より早い件数=("販社より早い", lambda x: (x=="○").sum()))
```
合成データの例では N=5→10→20 で「販社より早い」が24→32→38件と伸び、
N=40で無制限と同じ41件に飽和した。**飽和点の手前＝追加1件のNで勝ち件数が
伸びなくなる点**が、諦めラインの候補になる。高影響インシデントだけで層別して
同じ表を出せば「高影響は全部拾える最小N」も分かる。

### capped_月次負荷.csv（1行=月×N）
発火候補 / 処理 / 積み残し / 再評価 / 保留継続 / 二軍水準（当月ウォッチ線以上で
インボックス外の件数）/ 対象単位数。積み残しが単調増加し続けるNは予算不足の兆候
（本物でない滞留はSが伸びず沈むので、健全なら積み残しは頭打ちになる）。

## 仕様上の注意
- **持ち越し＝再競争**：注入しないだけ。Sは積み上がったままなので本物は翌月以降
  注目度順に勝ち上がる（reset_after_alarm=False の設計思想どおり）。
- **キャップは「発火」のみに適用**：再評価・保留継続は正規triageより軽い確認作業
  なのでNに数えない。件数は月次表に別列で出るので、負荷全体はそちらで見る。
- 注入処置は「ノイズ」「対策中」のみ（base不変の前提。fast版と同じガード）。
- top_n=None は simulate_historical_load_fast と月次内訳・台帳が完全一致
  （verify_capped.py 検証1）。増分キャッシュは毎月フルリプレイの参照実装と
  N=1/2/無制限×2処置すべてで一致（検証2）。
- 検知率がNに依らず一定なのは仕様（キャップは検知でなく処理タイミングを変える）。
  検知率<100%のラベルは、そもそも操作点で拾えていない＝バックテスト側の話。
- 二軍ウォッチはこのシミュレータでは「線を跨いだ月の記録」まで。実運用の目視は
  tableau_監視テーブルの注目度列に watch_frac の参照線を引くだけで始められる。

## Nスイープが遅い場合
run_sweep は静的前処理と初期リプレイを全Nで共有済み。それでも遅ければ
`start_ym` で立ち上がり初月の塊を外す、`n_list` を絞る、の順で。
collect_details=True は必要な時だけ（月×単位の内訳行を作るぶん遅くなる）。
