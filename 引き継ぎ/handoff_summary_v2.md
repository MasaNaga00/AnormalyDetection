# 引き継ぎサマリ v2 - 部品出庫率による出庫状態の異常検知

別チャットで検討を継続するための引き継ぎ資料。新しいチャットの冒頭にこの内容を貼り、
あわせて成果物一式(7章のファイル)をアップロードすれば、文脈を再説明せずに続きから
始められる。初版サマリ(260608)以降の進展(スパイク検知・安定化前モード・自動振り分け・
運用ランナー・ドキュメント)を反映した最新版。

---

## 0. これは何の検討か

修理部品の出庫データから、製品(機種)の異常な故障増加を**早期に**検知する仕組みの設計。
現場の経験則「同じタイプの故障が一定割合を超えたら確認した方がよい」を、自動的・定量的な
監視に置き換えることが目的。設計はほぼ固まり、検証ツール群・運用ランナー・設計ドキュメント・
使用方法/検証手順書まで作成済み(7章参照)。コンテナ内で全ツールの動作確認済み。

---

## 1. 制約・前提(依頼者の状況)

- 主にPythonで作業。可視化・共有はTableau Desktop。
- **会社データは社外に出せない**。実データもラベルも共有不可。検証は依頼者環境で完結。
- 部品は**低頻度**(多くの部番で月0〜数個)。
- 製造は継続中。**スパイク(突発増)も実データには存在し、件数が少ないものも検出が必要**
  (初版では「スパイクは対象外」としていたが、依頼者の指摘で方針変更。下記3・4参照)。
- `SF-コード`(類似部品の横断グループ)あたりの機種数は **2〜7機種**(少ないと2〜3)。
- **機種ランク差**がある。高ランク(高価格)機種は元々修理率が高い。
- 依存ライブラリは numpy / pandas のみ(scipy不要)。Python 3.10+。

---

## 2. データ

2つのデータマート。一意キーは「事業コード × 開発コード × 部番 × 販社 × 年月」(縦長月次パネル)。

- **datamart_A**: 使用数系に加え販売台数系(月次/累積販売台数・使用率)を持つ。率が出る主データ。
- **Datamart_B**: 使用数のみ。販売台数系なし。

主要カラム: `経過月`(発売からの経過、期待値の時間軸) / `累積販売台数`(保有台数近似、分母) /
`SF-コード`(横断グループ、安定化前モードの集団キーに使用) / `月次使用数`(監視する生カウント) /
`ランク`(価格帯、安定化前モードで集団を分けるキー)。

---

## 3. 検知のスコープ(現在の合意)

- **持続的な上振れ(ドリフト)**: 安定期の機種が劣化し始める緩やかな上昇。設計/ロット起因。**主**。
- **突発的な上振れ(スパイク)**: ロット不良流出などの単月〜数か月の急増。**初版から方針変更して対象に追加**。
- **初期不良(安定化前)**: 発売間もない機種の異常。先行機種カーブで監視。**優先度は中**(補助的)。

→ 結果として **ドリフト・スパイクを両エンジン併走**で、**安定期・安定化前を自動振り分け**で
カバーする構成になった。

---

## 4. 設計判断とその理由(再検討しない前提)

初版からの判断(維持):
1. **累積使用率の固定しきい値1本 → 早期検知エンジンには不採用**: 遅行指標で手遅れ。経過月と交絡。
2. **SF-コードで束ねて監視 → 却下**: 機種単位の異変が消える。
3. **SF-コードで他機種と横並び比較 → 却下**: ランク差で物差しが歪む、機種数が少なすぎる。
4. **自己基準のドリフト検知 → 採用**: 各機種を自身の過去と比較。ランク問題が消え機種数に依存しない。
5. **PCAでの機種横断 → 却下**: サンプル不足・ランク差が第1主成分を支配。横断プーリングが要るなら
   階層ベイズ・ポアソン部分プーリング(これは安定化前モードで実際に採用)。

このチャットで追加・変更した判断:
6. **スパイク検知を追加(初版の「対象外」を撤回)**: 実データには低頻度のスパイクが存在し検出必要。
   ドリフト(CUSUM)と**得意領域が逆**なので併走させる(combined Shewhart-CUSUM)。
7. **スパイク検定は条件付き二項検定**: 単純ポアソン検定(lambda0を真値扱い)は、12か月の短い
   ベースラインでlambda0を過小推定した単位で**誤報を量産**した(理論値の数十倍)。ベースライン窓の
   カウントとの「2つのポアソンレート比較の正確検定(条件付き二項)」に置換し、推定不確実性を
   織り込んで誤報を理論値どおりに抑えた。**安定化前モードでは**自己ベースラインが無いので各月の
   直接ポアソン検定にフォールバック。
8. **安定化前は時変カーブ lambda0(t) で監視**: 固定lambda0をスカラ→配列に拡張。故障率の
   「初期高→減衰→安定」を先行機種から推定したカーブで表す。エンジン側の数式変更は小さい
   (k_t = (R-1)·lambda0_t·fleet_t / ln R)。
9. **カーブ推定は経験ベイズ Gamma-Poisson 部分プーリング**: MCMC不要、numpy/scipyのみ。薄い機種を
   集団平均へ縮約。ランク差は集団キー(SF-コード×ランク)を分けて対処。早期の期待値ゼロ連発による
   p=0偽信号を、経過月方向の平滑化とゼロ下限で抑制。**推定カーブで新機種自身のスケールを
   合わせ直さない**(異常をカーブが吸収するため。移動平均lambda0問題と同根)。
10. **安定期/安定化前の自動振り分け**: 各単位の最大経過月で判定。
    `最大経過月 >= stable_start_m + baseline_len - 1` なら安定期、未満は安定化前候補、
    先行機種が `min_leaders` 未満なら監視保留。

## 5. 最終設計(骨格)

### 監視指標
1台あたり月次故障ペース。生レートを正規分布扱いせず、月次使用数(離散カウント)を
時変平均ポアソン過程として監視。

### エンジン1: ドリフト検知(時変平均ポアソン上側CUSUM)
- 平常レート `lambda0` を安定期初期で推定し**固定**(移動平均更新はドリフトを吸うので不可)。
- 期待故障数 `mu0_t = lambda0 × 累積販売台数_t`。保有台数増を自動吸収しレート上昇分のみ反応。
- 上側CUSUM: `k_t = (R-1)·lambda0·fleet_t / ln R`、`S_t = max(0, S_{t-1} + 使用数_t − k_t)`、
  アラート `S_t ≥ h`。パラメータ `R`(悪化倍率) と `h`(しきい値)。
- **lambda0はスカラ/配列両対応**: 安定化前は配列 lambda0_t(カーブ)を渡す。

### エンジン2: スパイク検知(Shewhart相当)
- 単月検定: 当月カウントの上側裾確率 p_t を出し、`p_t ≤ alpha_spike かつ 使用数 ≥ min_count` で点灯。
- バーストウィンドウ検定(任意): 直近W=2〜3か月の合算で同じ検定。月境界割れや中間レンジの急増担当。
- **安定期は条件付き二項検定**(ベースライン窓 C,E との比較)、**安定化前は直接ポアソン検定**。
- min_count ガードで「期待値極小のとき数個で鳴る」を防ぐ。

### 基準モードの自動振り分け
- **安定期モード**: 自己ベースラインから固定lambda0。履歴の揃った機種。
- **安定化前モード**: 同集団(SF-コード×ランク)の先行機種から推定したカーブ lambda0(t)。若い機種。
- **監視保留**: 安定期に達せず先行機種カーブも作れない単位。

### 監視単位: 二段構え
1. **機種×部番(販社合算)**: 感度優先の主監視。安定化前モードもこちらで実施。
2. **機種×部番×販社**: 偏りの所在を切り分ける副監視。

### 妥当性ウィンドウ
`累積販売台数` は退役が進む末期に過大化し、上側CUSUMでは本物の上昇を**マスク**する方向に効く。
監視を `monitor_end_m` まで(退役顕著化前の安定期)に限定。末期は守備範囲外。

### Tableau連携
運用ランナーが縦長CSVを出力。`期待故障数`・`CUSUM`・`しきい値h`・`月次p値`・`アラート種別`・
`総合アラート`・`基準モード` 等を含む。実績vs期待・CUSUMvsしきい値・worklist(p値昇順)を
同一データソースで描ける。

## 6. 決定済み事項(再検討不要)

- 分母(累積販売台数)の稼働台数近似問題 → 妥当性ウィンドウで対処。末期は対象外。
- **B限定部品(率が出せない)→ 監視対象外**。
- スパイクとドリフトは**両方常時稼働**(補完関係、片方では取りこぼす)。
- 検知タイプのチューニングは独立(ドリフト=R,h / スパイク=alpha_spike,min_count,burst_window)。
- グリッドサーチには**全タイプのラベルを渡す**(一部だけだと誤報が過大計上)。検知率は power_<タイプ> 列で読む。
- ベースライン再推定は**人間の明示判断**(自動再推定は問題をマスクする)。

## 7. 作成済みの成果物(次チャットにアップロードするもの)

すべてローカル完結・ネットワーク通信なし。

**コード(エンジン・検証・運用)**
- `cusum_monitor.py`: 検知エンジン本体。CUSUM(ドリフト)+ スパイク検定、安定期/安定化前 両対応。
  lambda0スカラ/配列両対応。条件付き二項検定。
- `earlylife_baseline.py`: 安定化前モード用。先行機種から期待カーブ lambda0(t) を経験ベイズ
  (Gamma-Poisson部分プーリング)で推定。平滑化・ゼロ下限つき。
- `backtest.py`: 検証ハーネス。evaluate / grid_search / spike_grid_search / suggest_operating_point。
  検知率・検知遅れ・誤報率、タイプ別検知率(power_drift/power_spike)。
- `run_monthly.py`: 毎月の運用ランナー。前処理(月次0補完・販社合算)→自動振り分け→検知→CSV出力。
  CONFIG(安定期操作点) と EARLYLIFE(安定化前操作点・集団キー) を冒頭で設定。

**デモ・データ生成(疑似データ、社外共有可)**
- `synthetic_demo.py`: 安定期エンジンの動作確認(ドリフト+スパイク併走、タイプ別検知率)。
- `earlylife_demo.py`: 安定化前モードの動作確認(正常な新機種は鳴らず、異常な新機種は早期点灯)。
- `make_mixed_input.py`: 自動振り分けデモ用の混在疑似データ(安定期+安定化前、SF-コード・ランク付き)。

**ドキュメント**
- `使用方法と検証手順.md`: 使い方・検証手順の正式版。入力形式、クイックスタート、運用ランナー、
  検証/チューニング、カーブ推定、Tableau連携、パラメータ早見表。記載コード例は動作確認済み。
- (初版から継続) `部品出庫_異常検知_設計ドキュメント.md`、初版引き継ぎサマリ等があれば併せて。

### 検証の考え方(変わらず)
依頼者環境でラベル(過去事例の「確認すべきだった経過月」、可能なら drift/spike のタイプ列)を入力 →
各単位を監視 → 検知遅れ(負=早い)・検知率・誤報率を算出 → (R,h) と (alpha_spike,min_count) を
独立にグリッド化 → 誤報予算内で最も早く・多く検知できる操作点を選ぶ → run_monthly.py の CONFIG に反映。

## 8. 次にやること / 未決事項

実装はほぼ揃った。残るのは実データ投入と運用調整。

- **実データでの操作点決定**: backtest.py でグリッドサーチを実行し、現場感覚に合う
  (R,h) と (alpha_spike,min_count,burst_window) を確定。ラベルが少なければ時系列分割・
  leave-one-out で過剰適合回避。確定値を run_monthly.py の CONFIG/EARLYLIFE に反映。
- **振り分け規模の把握**: 実データで「stable_start_m+baseline_len 未満の若い機種がどれだけあるか」
  「先行機種が min_leaders 以上揃う集団がどれだけあるか」を確認し、安定化前モードの実効カバー率を見る。
- **運用方針の確定**: 「誤報を絞る」か「空振り覚悟で早める」か。安定期と安定化前で別感度にするか。
- **販社別監視の閾値**: 合算側と別に感度を持つか。安定化前モードを販社別でも有効にするかは要検討
  (現状は主監視=合算側のみ安定化前を実施)。
- **アラートライフサイクル管理**: 状態機械+ランクworklistの設計は合意済み。Tableauダッシュボードへの
  `アラート種別`・`月次p値`・`基準モード` の組み込み、再アラート条件メニューの実装が次段階。
- **(任意)ドキュメントのWord化**(レビュー配布用)。
- **(任意)カーブ推定の高度化**: 先行機種が極端に少ない集団向けに、ランクを共変量にした
  対数線形モデル等。現状は「集団を分ける」素直な方法に留めている。
