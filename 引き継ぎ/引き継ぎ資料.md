## 1. プロジェクト概要
- **目的**：修理部品の出庫異常を自動検知。監視単位＝**機種×部番（販社合算）**。
- **2種類の異常**：(a) 安定期に徐々に進む故障率上昇＝**ドリフト**、(b) 突発的な**スパイク**。両方を自動検出。
- **成功の定義**：月間アラート予算（チームのtriage能力）の範囲内で、真の異常を早期・確実に捉える。
- **2トラック構成**：
  - 固定しきい値・累積使用率モニター（`state_logic.py`、別途稼働中）。
  - **本トラック＝CUSUM/Shewhart**（このプロジェクト）。

## 2. システム全体像
3層構造。検知の数式は既存エンジンに委譲し、その上に人間レビュー運用を載せる。

```
[パネル(機種×部番×販社×月)] + [台帳(人間の判断, マスタ)]
        │
        ▼  state_logic_cusum.py（本体）
  ┌─ モード分類（安定期 / 安定化前 / 安定化前(保留)）
  ├─ ベースライン: 安定期=自己, 安定化前=先行機種カーブ lambda0(t)
  ├─ 検知（委譲）:
  │     ドリフト → cusum_monitor.poisson_cusum（時変平均ポアソン上側CUSUM）
  │     スパイク → cusum_monitor.spike_test（安定期=条件付き二項 / 安定化前=直接ポアソン）
  │     カーブ  → earlylife_baseline.estimate_earlylife_curve + attach_curve_to_unit
  ├─ リセット注入（台帳のリセット点でセグメント分割, 各セグメントS=0から）
  ├─ 状態解決（正常/監視中/要確認/要確認保留/対策効果確認中/終了）
  └─ 出力: 要確認インボックス.csv（triage用）, tableau_監視テーブル.csv（可視化用）
```

## 4. 確定した設計判断（再検討不要・根拠つき）

### 4.1 リセットの噛み合わせ
`poisson_cusum` は一括計算で既定 `reset_after_alarm=True`（発火月に自動S=0）。本運用は **`reset_after_alarm=False` に固定**し、リセットは台帳由来（reset-after-M）だけにする。
- 根拠：自動リセットだとSが鋸歯状になり、未対応ドリフトが谷間で見落とされる。自動を切れば未対応の問題はSが積み上がり注目度が上がり続け、worklist上位に来る（triage圧が正しくかかる）。
- 実装：台帳のリセット点でタイムラインを**セグメント分割**し、各セグメントをS=0から `poisson_cusum` で計算。reset-after-M＝判定月Mは発火としてセグメント末尾に残し、翌月から新セグメント。
- 好都合：リセットを伴う処置区分はすべてセグメント境界になり、lambda0/C/E や R/h を変える処置（新常態受容・感度上書き）は同時にリセットも起こす → **1セグメント内でこれらは一定**。

### 4.2 スパイク検定＝条件付き二項
安定期はベースライン窓の `(C, E)=(使用数合計, 露出合計)` を `spike_test` に渡し、2つのポアソンレート比較の正確検定（条件付き二項）にする。
- 根拠：短い（12ヶ月程度）ベースラインだと lambda0 を過小推定し、素朴ポアソン裾だと誤報が膨らむ。条件付き二項は推定不確実性を織り込み、ベースラインが薄いほど保守側に開く（決して素朴より鳴りやすくならない）。
- 安定化前は配列 lambda0 を渡すと `spike_test` が自動で直接ポアソンに切り替わる（自己ベースライン C,E が無いため）。

### 4.3 安定化前カーブ
- **leaders＝安定期単位のみ**。監視対象（安定化前）は定義上含まれない＝**スケールが先行機種側で固定**（自身の異常をカーブが吸わない）。
- **監視保留**：先行機種が `min_leaders` 未満の集団は lambda0 を**全0**にして監視不能＝鳴らさない（floorを当てない）。安定期レートのプール平均で代用しない（初期高を無視して誤報を量産するため）。
- **ランク階層フォールバック**（`earlylife_group_levels`、既定 `[["biz","sf","rank"],["biz","sf"]]`）：細→粗。キーが揃い先行機種≥min_leaders の最初の階層を採用。ランクが一部の機種にしか無くても、ランクのある集団はランク単位、薄い/欠損はSF単位に自動で落ちる。ランク列が全く無ければランク階層は自動スキップ。

### 4.4 PCA 不採用（過去に確定済み）
SF-コード内の機種数は2〜7と少なく、ランク差が第1主成分を支配するため安定したPCAは困難。自己参照ドリフト検知（自機種の安定期実績との比較）が正しい主軸。

### 4.5 台帳設計
Excel単一ファイルをマスタ、追記型。CSV並列維持は不採用。人間判断を再アラートワークフローに残す（自動化しない）。状態は保存せず毎回パネル＋台帳から再生（asof再現性の根拠）。

### 4.6 操作点バックテスト
ドリフトとスパイクは独立検出器なので別グリッド。誤報はラベル不要（実データ＝ほぼ正常）、検出力は合成注入（＋あれば実ラベル）。ドリフトのARL0/遅れ測定は各アラートを離散イベントにするため `reset_after_alarm=True`（測定の流儀。本番のリセットは台帳由来）。**月間アラート予算**＝チームのtriage能力で、艦隊全体の月間誤報件数（=誤報率×監視単位数）で線引きする。

## 5. 重要な学び・落とし穴
- **lambda0_floor を保留単位に当てない**：floor(1e-6)を当てると mu0 が極小だが非ゼロになり、使用数が出ると p≈0 で誤発火する。保留は生の0を渡して監視不能にする。
- **デモ/合成データの作り方**：先行機種にも初期不良カーブ（初期高→減衰）を持たせること。平坦レートで作ると、正常な新機種が「初期が高い」せいで異常判定される（フェーズ3で踏んだ）。
- **asof再現性の範囲**：安定期は完全再現。安定化前は**先行機種が成熟していれば**完全再現（若い経過月のカーブは古い＝打ち切りに残るデータから作られる）。先行機種自体が若く同時進化している場合のみ、データ増加でカーブが後から精緻化されて過去 lambda0(t) がわずかに動く（ベースライン再推定と同根）。厳密な監査再現が要るならasof時点のカーブをスナップショット固定。
- **spike_test のバースト**：各セグメント先頭 W-1 月は窓が揃わず NaN（リセット直後はバースト窓も再開）。
- **配列 lambda0 と行の対応**：`replay_unit` に渡す前に unit を ym 昇順・連番indexにそろえる（`evaluate_units` 内で実施済み）。

## 6. CONFIG リファレンス（主要）
`state_logic_cusum.CONFIG` をコピーして使う。
- 入出力：`panel_path, panel_sheet, ledger_path, ledger_sheet, out_inbox, out_tableau`
- 列名：`cols`（biz/dev/part/dist/ym/elapsed/monthly_use/cum_sales/sf/rank）
- ドリフト：`R=2.0, h=5.0`（**バックテストで確定して入れる**）
- スパイク：`alpha_spike=0.005, min_count=3, burst_window=0, alpha_burst=None`
- リセット：`reset_after_alarm=False`（**変更しない**）
- ベースライン/レンジ：`stable_start_m=4, baseline_len=12, monitor_end_m=60, lambda0_floor=1e-6`
- 安定化前：`min_leaders=3, earlylife_group_levels=[["biz","sf","rank"],["biz","sf"]], earlylife_smooth_window=3, earlylife_rate_floor_frac=0.1`
- その他：`fill_zero_months=True, asof_ym=None, machine_all_part_token="機種全体"`

## 8. これからやること（要約。詳細は `これからやること_チェックリスト.md`）
- **A. 操作点の確定（最優先）**：月間アラート予算決め → `backtest_cusum.run_backtest` を実データで → 表から R,h,alpha_spike,min_count,burst_window を選び CONFIG へ。
- **B. 初回本番立ち上げ**：パネル・台帳用意 → `run(cfg)` → 初回インボックスをレビュー。
- **C. 月次運用の定着**：最新パネル→run→インボックスレビュー→台帳追記、の月次サイクル。
- **D. Tableauダッシュボード（CUSUM版）**：`tableau_監視テーブル_cusum.csv` を入力にワークリスト/ドリルダウン/ヒートマップ/診断。
- **E. 精緻化**：先行機種から異常期間（対策中/監視終了）を除外、ランク段階導入、必要ならカーブのスナップショット固定。
- **F. 中長期**：固定しきい値トラックとの並行運用統合、販社別（機種×部番×販社）二次監視トラック（設計済み・未実装）。
