## 1. このセッションで確定した一連の運用フロー

実データが来たとき、この順で手を動かせば立ち上げられる、という流れをテストデータで実証した。

### 手順A：実データの投入前チェック（`_prepare_panel` に通す前）
1. 必須列の存在・欠損・型・重複を確認。
2. **年月がハイフン区切り（"2023-02"）でも問題なし** … `to_yyyymm()` が数字抽出で自動変換。
   ただし月がゼロ埋めされていない表記（"2023-2"）が混じると5桁誤変換になるので、
   文字列長のばらつきだけは確認する。
3. **累積販売台数の非単調**は、まず数値YYYYMMでソートし直してから件数を数える
   （文字列ソート起因の見かけ上の逆転が大半）。本物の逆転が残ったら中身を目視。
   実データでは「修正による小さな逆転」と判断できたので、機種×部番×販社ごとに
   `cummax()` で単調非減少に補正してから使う（露出=母数の一時減少がlambda0を歪めるため）。

```python
df["年月"] = df["年月"].map(s.to_yyyymm)
df = df.sort_values(["事業コード","開発コード","部番","販社","年月"])
df["累積販売台数"] = df.groupby(["事業コード","開発コード","部番","販社"])["累積販売台数"].cummax()
```

### 手順B：操作点決め（バックテスト）
1. `run_backtest("panel.csv", cfg=cfg, labels_path="labels.csv")` を実行。
   → `backtest_drift.csv` / `backtest_spike.csv` / `backtest_labeled.csv` が出る。
2. `backtest_drift.csv` を眺めて h↑で「月間誤報件数↓・遅れ↑」のトレードオフを掴む。
   実データは監視単位数が多い（今回3086）ので**月間誤報件数＝誤報率×単位数**が跳ねる点に注意。
3. `recommend_drift(tbl, max_alarms_per_month=予算, factor=最低捕捉倍率)` /
   `recommend_spike(...)` で予算内候補を検出率→遅れ→誤報の順に絞る。
4. **`backtest_labeled.csv` で既知インシデントを取りこぼしていないか必ず確認**。
   候補の中に取りこぼす設定があれば除外（テストではR≥2.0がS04を落とした）。
5. 1点選んで `state_logic_cusum.CONFIG` に反映。日付つきJSONで確定版を残すと監査に効く。

### 手順C：過去データでの月次triage負荷の実測（新規ツール）
本番前に「毎月何件チェックが必要だったか」を実データで実測する。バックテストの合成注入より
実態に近いので、月間アラート予算の最終確認に使う。

```python
from simulate_historical_load import simulate_monthly_load, compare_dispositions
cmp = compare_dispositions(units, cfg)                     # 楽観(ノイズ)〜現実的(対策中)の幅
monthly, details, _ = simulate_monthly_load(units, cfg, assumed_disposition="対策中")
```
- 「発火したらその場で台帳記録して再着火抑制」を月送りで再現。
- `assumed_disposition` で消し込み方を変えられる：`"ノイズ"`（記録して終わり＝楽観）／
  `"対策中"`（reeval_months後に再評価が発生＝現実的）。両方で幅を見るのが安全。
- 立ち上がり初月は積み残しが塊で出やすい（今まで見ていなかった分の答え合わせ）ので慌てない。

### 手順D：本番初回投入
1. CONFIG確定値をJSONで固定 → `verify_phase3.py` で asof再現性(差0)を最終確認。
2. **空台帳で1回 `run()`**（台帳テンプレートの「台帳」シートが0行の状態）。今の溜まりを一気に洗い出す特別回。
3. インボックスを注目度順にレビュー。件数が予算超過なら上位着手＋残りは「保留」で明示先送り。

### 手順E：月次運用サイクル
最新パネル整形 → `run()` → インボックスをレビュー → 台帳（Excel）に処置区分追記 →
`tableau_監視テーブル.csv` をTableauで更新共有。Tableauはデータソースを固定パスにしておけば
毎月は更新（Ctrl+Alt+F5）だけで済む。

## 3. このセッションの修正・新規ファイル詳細

### 3.1 `backtest_cusum.py`（修正）
- **`measure_labeled` を早期検知対応に**：発生年月（＝販社報告月）の `lookback_m`（既定6）ヶ月前まで
  遡ってアラームを探す。報告より早い発火は**遅れ月がマイナス**になる（＝早期検知の成功、失敗ではない）。
  旧版は発生年月「以降」しか見ず負値が構造的に出なかった＝早期検知を測れていなかった。
- **出力にパラメータ列を追加**（detector/R/h/alpha/min_count/burst_window/lookback_m）。
  CSV単体でどの設定の結果か分かる。
- **`measure_labeled_grid` を新設**：グリッド全点でラベル評価を縦積み＋サマリ表示。
- `run_backtest` は `backtest_labeled.csv` をグリッド全点で出力するよう変更。
- 検証：早期検知（真の変化点を報告4ヶ月前に置く）で遅れ月=−4 を確認。内蔵デモも回帰なし。

### 3.2 `state_logic_cusum.py`（**性能修正・要差し替え**）
- **`resolve_state` のO(n²)を解消**。旧版は itertuples ループ内で `rep[rep["ym"]==ym]` と
  rep全体を毎月再フィルタしており、監視月数の二乗でコストが増えていた。
  呼び出し側が既に持つ `r.total_alert` を直接渡す形に変更（ロジック不変）。
- **効果**：800機種で 56.7秒→10.5秒（約5.4倍）。実データ3086機種は280秒→**50〜55秒見込み**。
- **検証**：オリジナル版と修正版で table のセル単位差分0・インボックス完全一致を確認済み。
- メモリは元々121MBで問題なし＝遅さの原因はメモリではなく計算量だった。

### 3.3 `evaluate_units_progress.py`（新規・本体不変更のラッパー）
- `evaluate_units_stream(...)`：進捗表示（残り時間目安つき）＋ chunk_size 機種ごとの
  ディスク逐次書き出し（メモリ節約）＋ `use_multiprocessing=True` で機種単位の並列化。
- 内部関数をそのまま呼ぶだけなので `evaluate_units` と結果完全一致（逐次版・並列版とも差分0を検証）。
- 3.2の修正だけで足りるかを先に見て、まだ遅ければこれで並列化を上乗せする、という順で使う。

### 3.4 `simulate_historical_load.py`（新規）
- 手順Cのツール。`simulate_monthly_load` / `compare_dispositions`。
- 空inbox月のガード修正済み。

## 4. 未対処の設計課題（実データ運用前に方針決めが要る）

### 4.1 「スパイク確認」がCUSUMドリフトの再アラートを抑制しない（既知・最重要）
- 大きな単月スパイクはCUSUM（ドリフト検出器）側も一緒にSを積み増す。「スパイク確認」は
  設計上リセットなしなので、Sが高止まりして翌月以降ドリフトとして再点灯し続ける。
- テストデータのS03で再現（1回目run→スパイク確認記録→2回目runでdrift再点灯）。
- 緩和オプション（前回提示済み、未決）：
  (1) スパイク確認記録時、当月にdrift併発ならdrift側だけ慣習的にリセットする専用区分を作る。
  (2) 運用ルール化：drift+spike同時月は原因切り分けて「感度上書き」か「ノイズ」等リセット付き区分を選ぶ。
  (3) 現状維持：再点灯したら「累積の名残」と判断して再処置（triage件数は増える）。
- **実データでも大ロット不良が単月スパイクで出れば起こる**。運用ガイドに注記＋方針決定が必要。
