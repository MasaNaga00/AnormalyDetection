# 検証・チューニング手順書

社内のラベル付きデータ（過去のアラート相当事例）を**社外に出さずに**、検知ロジックの性能評価と感度パラメータの調整を行うための手順。すべて御社環境のPythonで完結する。

同梱物:
- `cusum_monitor.py` … 監視エンジン（時変平均ポアソンCUSUM）
- `backtest.py` … 検証ハーネス（検知性能の計測・グリッドサーチ）
- `synthetic_demo.py` … 疑似データでの動作確認デモ兼使用例
- 本手順書

## 1. 入力データの形式（実データ）

### 1.1 監視対象データ `df`

`datamart_A` から、以下の列を持つ縦長 DataFrame を用意する（列名はそのまま）。

| 列 | 必須 | 内容 |
|---|---|---|
| 事業コード / 開発コード / 部番 / 販社 | ○ | 監視単位キー |
| 経過月 | ○ | 発売からの経過月。監視・ベースラインの時間軸 |
| 月次使用数 | ○ | 監視する生カウント |
| 累積販売台数 | ○ | 期待値の分母（保有台数近似）|
| 年月 | 任意 | 表示用 |

注意:
- **B限定部品（販売台数が取れず率が出せないもの）は対象外**。`df` には含めない。
- 使用がなかった月の行が欠落していると経過月が飛ぶ。月次0の行は0として補完しておくこと（CUSUMは0の月も「上振れしていない証拠」として使う）。

### 1.2 ラベル（過去の異常事例）`labels`

1行 = 1異常事例。社内で把握している「ここで確認すべきだった」時点を経過月で与える。

| 列 | 内容 |
|---|---|
| 事業コード / 開発コード / 部番 [/ 販社] | 対象単位のキー |
| ラベル経過月 | その単位で異常が発覚／確認された経過月 |

ラベルが販社別でなく機種単位なら、販社合算監視（後述）と組み合わせる。

## 2. 実データでの実行

```python
import pandas as pd
import cusum_monitor as cm
import backtest as bt

df = pd.read_csv("datamart_A_前処理済み.csv")     # 1.1 の形式
labels = pd.read_csv("labels.csv")                # 1.2 の形式

# 販社合算で監視する場合(感度優先の主監視)
df_agg = cm.aggregate_over_sha(df)
unit_keys = cm.UNIT_KEYS_AGG_SHA                  # 機種×部番

# 監視レンジとベースライン(3章・4章で決める)
common = dict(stable_start_m=36, baseline_len=12, monitor_end_m=90)

# グリッドサーチ
grid = bt.grid_search(
    df_agg, labels, unit_keys, **common,
    R_grid=[1.5, 2.0, 2.5, 3.0],     # 悪化倍率(検知したいレート上昇)
    h_grid=[5, 8, 12, 18, 25],       # しきい値
)
print(grid[["R","h","power","median_delay","fa_per_unit_year"]])

# 誤報予算と最低検知率を満たす操作点を早期順に
op = bt.suggest_operating_point(grid, max_fa_per_unit_year=0.02, min_power=0.8)
print(op)
```

販社別の副監視は `unit_keys = cm.UNIT_KEYS_WITH_SHA`、`df`（合算前）を渡す。

## 3. 妥当性ウィンドウ（分母＝稼働台数問題への対処）

`累積販売台数` は実稼働台数の近似だが、年数が経つと退役で実稼働が減り、入庫も減る。一方 累積販売台数 は増え続ける。この乖離は上側CUSUMでは**誤報を増やす方向ではなく、本物の上昇を見えにくくする（マスクする）方向**に効く（期待値が過大になり、実績との差が下にずれるため）。

対処は、近似が成立する経過月レンジに監視を限定すること。`monitor_end_m` がその上限（妥当性ウィンドウの末尾）。

決め方の目安:
- 主ターゲットの安定期は退役が顕著になる前なので、安定期の範囲とおおむね一致させる。
- データから決めるなら、`累積販売台数` の月次増分（＝月次販売台数）が十分に立っている、あるいは累積使用数がまだ伸びている区間を選ぶ。増分がほぼ0になり累積使用数も頭打ちした以降は監視対象から外す。
- 事業コード／部品種ごとに退役の速さが違えば、`monitor_end_m` を分けて設定する。

`stable_start_m` は安定期の入口、`baseline_len` は平常レート推定に使う先頭月数。短すぎると lambda0 が不安定、長すぎるとベースライン期間に異常が混入するリスク。12か月前後から始め、感度を見ながら調整する。

## 4. 出力指標の読み方

| 指標 | 意味 | 望ましい方向 |
|---|---|---|
| `power` | ラベル事例のうち検知できた割合 | 高い |
| `median_delay` | 検知遅れの中央値（月）。`初アラート経過月 − ラベル経過月` | **負**（人手の発覚より早い）か、小さい正 |
| `fa_per_unit_year` | 平常単位での 誤報数 ÷ 監視単位・年 | 低い（運用で許容できる範囲）|

`delay` の符号: 負なら人手より早く検知＝早期検知として成功。正なら遅れて検知。検知成功とみなす窓は `evaluate` の `early_margin`／`late_margin` で調整できる。

操作点の選び方: まず運用上許容できる誤報予算（例: 監視単位・年あたり0.02件）を決め、それを満たす中で `power` が高く `median_delay` が小さい（早い）`(R, h)` を選ぶ。**低頻度データでは「高検知率・低誤報・早期」を同時に満たせないのが普通**で、グリッドはそのトレードオフを可視化するためのもの。「2〜3か月遅れても誤報を絞る」か「空振り覚悟で早める」かの運用方針が、操作点の位置を決める。

## 5. パラメータをラベルに過剰適合させない

ラベル件数が少ない場合、グリッドで選んだ `(R, h)` がそのラベル集合にだけ都合よく合ってしまう恐れがある。対策:

- **時系列分割**: 古い時期のラベルでパラメータを決め、新しい時期のラベルで検証する。
- **leave-one-out**: ラベルを1件ずつ抜いて残りで決めたパラメータが、抜いた1件を検知できるか確認する。
- 事業コードや部品種でグループが分かれるなら、グループ横断で安定して効く `(R, h)` を選ぶ（特定グループだけに効く設定は避ける）。

`backtest.py` は `labels` を絞って渡せば任意の分割で再評価できる。

## 6. 結果をTableauへ

操作点が決まったら、その `(R, h)` で `cm.monitor_all(...)` を全期間に対して実行し、返り値（`期待故障数`・`CUSUM`・`しきい値h`・`アラート` 等を含む縦長テーブル）をCSV出力する。これが設計ドキュメント8章のTableau入力テーブルになる。Tableauでは実績 vs 期待故障数の対比、CUSUM vs しきい値、アラート点のハイライトを同一データソースで描ける。

## 7. プライバシーについて

本ツールはネットワーク通信を行わず、入力データ・ラベル・結果はすべてローカルファイルとメモリ内に留まる。性能評価・パラメータ調整は御社環境内で完結し、データやラベルを外部に共有する必要はない。共有が必要になるのは、必要なら出力サマリ（検知率・遅れ・誤報率などの集計値）のみで、生データや事例の中身ではない。
