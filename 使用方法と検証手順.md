# 部品出庫 異常検知システム — 使用方法と検証手順

修理部品の出庫データから、製品(機種)の異常な故障増加を早期に検知する仕組みの
使用方法・検証手順書。設計の背景・判断理由は別資料(設計ドキュメント・引き継ぎサマリ)
を参照し、本書は「どう動かすか」「どう検証するか」に絞る。

すべてのツールはネットワーク通信を行わず、データ・ラベル・結果はローカルで完結する。

---

## 0. 全体像

検知は2つのエンジンを同じ基準レート(lambda0)の上で併走させる。

| エンジン | 何を捉えるか | 得意な信号 |
|---|---|---|
| ドリフト検知 (CUSUM) | 小さいが持続する上昇 | 設計起因・部品ロット起因の緩やかな劣化 |
| スパイク検知 (Shewhart相当) | 大きく突発的な上昇 | ロット不良流出などの単月〜数か月の急増 |

さらに、機種の履歴の長さで監視の基準を自動で切り替える。

| 基準モード | 物差し | 対象 |
|---|---|---|
| 安定期モード | その機種自身の過去 (固定lambda0) | 履歴が十分に揃った機種 |
| 安定化前モード | 同じ集団の先行機種から推定したカーブ lambda0(t) | 発売間もない若い機種 |

毎月の運用は「データを入れる → 自動振り分け → 検知 → 結果CSV → Tableau」。
感度パラメータ(操作点)は、運用前に検証ハーネスで1回決めておく。

## 1. ファイル構成

| ファイル | 役割 |
|---|---|
| `cusum_monitor.py` | 検知エンジン本体 (CUSUM + スパイク検定、安定期/安定化前 両対応) |
| `earlylife_baseline.py` | 安定化前モード用。先行機種から期待カーブ lambda0(t) を推定 |
| `backtest.py` | 検証ハーネス。過去事例ラベルに対する検知性能測定とパラメータ探索 |
| `run_monthly.py` | 毎月の運用ランナー。前処理→自動振り分け→検知→CSV出力 |
| `synthetic_demo.py` | 安定期エンジンの疑似データ動作確認デモ |
| `earlylife_demo.py` | 安定化前モードの疑似データ動作確認デモ |
| `make_mixed_input.py` | 自動振り分けデモ用の混在疑似データ生成 |

依存は Python 3.10+ / numpy / pandas のみ(scipyは不要)。

## 2. 入力データの形式

縦長(ロング)の月次パネル。1行 = 1監視単位の1か月。

必須列:

| 列名 | 意味 |
|---|---|
| `事業コード` | 監視単位キー |
| `開発コード` | 機種を表す開発コード。監視単位キー |
| `部番` | 部品番号。監視単位キー |
| `販社` | 販売会社。販社別監視のキー(合算監視では合算される) |
| `経過月` | 発売からの経過月数。期待値・ライフサイクルの時間軸 |
| `月次使用数` | その月の出庫カウント。監視する生の値 |
| `累積販売台数` | 保有台数の近似。期待故障数の分母 |

任意列(安定化前モードで使用):

| 列名 | 意味 |
|---|---|
| `SF-コード` | 類似部品の横断グループ。先行機種をプールする集団キー |
| `ランク` | 機種の価格帯/ランク。集団をランク内に分けるためのキー |
| `年月` | 実年月。監視は経過月で行うため必須ではない(あれば出力に残る) |

注意点:
- 月次0の行は欠落していてもよい(ランナーが補完する。下記4-1)。
- `累積販売台数` が取れない部品(B限定部品)は率が出せないため監視対象外。
- 列名が異なる場合は、各関数の `col_usage` / `col_fleet` / `col_keizoku` 引数で
  指定するか、読み込み直後にリネームする。

## 3. クイックスタート(疑似データで動作確認)

実データの前に、疑似データで仕組みが理論通り動くことを確認できる。

```bash
# 安定期エンジン(ドリフト+スパイク併走)の検知性能を見る
python synthetic_demo.py

# 安定化前モード(先行機種カーブによる初期不良検知)を見る
python earlylife_demo.py

# 自動振り分け(安定期/安定化前の混在)を運用ランナーで通す
python make_mixed_input.py      # 混在入力 mixed_input.csv を生成
python run_monthly.py           # 振り分けて検知、alerts_*.csv を出力
```

`synthetic_demo.py` の期待される出力(seed=1):
- ドリフト事例とスパイク事例の両方を含むラベルを生成。
- CUSUM単独だとスパイクを取りこぼすが、併走させると検知率が上がることが
  タイプ別内訳で確認できる。

`earlylife_demo.py` の期待される出力:
- 集団どおりの正常な新機種(NEW_OK)はアラート0。
- 初期から集団水準を超える異常機種(NEW_BAD)は早期に点灯。

## 4. 毎月の運用 (`run_monthly.py`)

### 4-1. 何をするか

1. 入力CSVを読む。
2. **前処理**: 月次0補完(欠落月を 使用数=0・台数は前方埋め で埋める)、
   販社合算(主監視用に機種×部番へ集約)。
3. **自動振り分け**: 各単位の最大経過月を見て、安定期/安定化前/監視保留に分ける。
4. **検知**: モードごとに適切な基準で CUSUM + スパイク検定を実行。
5. **出力**: 結果を縦長CSVに保存。サマリ(モード別単位数・当月点灯数)を表示。

### 4-2. 振り分けの規則

- **安定期モード**: `最大経過月 >= stable_start_m + baseline_len - 1`。
  自己ベースライン窓を取り切れる履歴がある単位。
- **安定化前モード**: 上記未満の若い単位のうち、同じ集団(`group_keys`)に
  先行機種が `min_leaders` 以上いるもの。先行機種は「安定期に分類された単位」を使う。
- **監視保留**: 安定期に達せず、先行機種カーブも作れない単位。今月は監視外。

### 4-3. 設定(ファイル冒頭の CONFIG / EARLYLIFE)

```python
CONFIG = dict(
    stable_start_m=36,   # 安定期の入口(経過月)
    baseline_len=12,     # ベースライン窓の長さ(月)
    monitor_end_m=90,    # 妥当性ウィンドウの末尾(これ以降は監視しない)
    R=2.0, h=8.0,                          # ドリフト側(CUSUM)操作点
    alpha_spike=0.001, min_count=3, burst_window=2,  # スパイク側操作点
)

EARLYLIFE = dict(
    enabled=True,
    group_keys=["事業コード", "SF-コード", "ランク"],  # 先行機種の集団キー
    curve_max_m=35,      # カーブ推定する経過月の上限
    min_leaders=2,       # カーブ推定に必要な先行機種の最低数
    R=2.0, h=6.0,                          # 安定化前側の操作点(別感度可)
    alpha_spike=0.005, min_count=3, burst_window=2,
)

INPUT_CSV = "real_like_input.csv"   # 入力ファイル名
```

操作点(R, h, alpha_spike など)は、運用前に検証ハーネス(5章)で決めた値を入れる。
振り分けの境界 `stable_start_m + baseline_len` を変えると安定期/安定化前の比率も動く。

### 4-4. 実行

```bash
python run_monthly.py
```

主監視(販社合算: 機種×部番)と副監視(販社別: 機種×部番×販社)の両方を実行し、
`alerts_agg_機種x部番.csv` と `alerts_sha_機種x部番x販社.csv` を出力する。

### 4-5. 出力CSVの列

| 列 | 意味 |
|---|---|
| (入力キー列) | 事業コード・開発コード・部番・販社・年月・経過月 |
| `月次使用数` / `累積販売台数` | 入力値(補完後) |
| `lambda0` | 基準レート。安定期は固定値、安定化前は各月のカーブ値 |
| `期待故障数` | lambda0 × 累積販売台数。その月に期待される故障数 |
| `参照値k` | CUSUMの時変参照値 |
| `CUSUM` | CUSUM統計量。`しきい値h` を超えるとドリフトアラート |
| `しきい値h` | CUSUMのしきい値 |
| `アラート` | ドリフト(CUSUM)アラートの真偽 |
| `月次p値` | 単月スパイク検定のp値。小さいほど深刻 |
| `スパイクアラート` | 単月スパイクアラートの真偽 |
| `バーストp値` / `バーストアラート` | 数か月合算のバースト検定 |
| `アラート種別` | "drift"/"spike"/"burst" を + 連結した種別 |
| `総合アラート` | いずれかのアラートが立ったか(全タイプのOR) |
| `基準モード` | "stable"(安定期) / "earlylife"(安定化前) |

### 4-6. 結果の読み方

- 毎月の要確認リストは「各単位の最新月で `総合アラート` が立っている単位」。
  延べアラート月数は時系列の履歴であり、worklistとは別。
- `月次p値` はそのまま深刻度の並び順に使える(小さいほど優先)。
- `アラート種別` で、持続的劣化(drift)か突発(spike/burst)かを区別できる。
- `基準モード` で安定期/安定化前を色分けすると、若い機種の信頼度の低さを
  踏まえた読み方ができる。

## 5. 運用前の検証とチューニング (`backtest.py`)

操作点(感度パラメータ)を、過去事例ラベルに対する検知性能から決める段階。
運用開始時に1回行い、その後は事例が増えたら見直す。

### 5-1. ラベルの形式

1行 = 1異常事例。

| 列 | 意味 |
|---|---|
| 監視単位キー | 事業コード・開発コード・部番(・販社) |
| `ラベル経過月` | 「ここで確認すべきだった」と分かっている経過月 |
| `タイプ` (任意) | drift / spike など事例の分類。あればタイプ別検知率が出る |

ラベルは社内で把握している過去の発覚/確認時点を入れる。社外共有不要。

### 5-2. 性能指標

| 指標 | 意味 |
|---|---|
| `power` | ラベル事例のうち検知できた割合(検知率) |
| `power_drift` / `power_spike` | タイプ別検知率(ラベルに「タイプ」列がある場合) |
| `median_delay` | 検知遅れの中央値(月)。負ほど人手より早い |
| `fa_per_unit_year` | 平常単位での誤報数 / 監視単位・年 |

検知判定: ラベル前 `early_margin` か月 ～ ラベル後 `late_margin` か月の窓内に
アラートがあれば検知成功。delay は窓内最初のアラートで計算。

### 5-3. パラメータ探索の流れ

ドリフト側 (R, h) とスパイク側 (alpha_spike, min_count) は狙う信号が違うため
独立にチューニングする。

```python
import pandas as pd
import cusum_monitor as cm
import backtest as bt

df = pd.read_csv("実データ.csv", encoding="utf-8-sig")
labels = pd.read_csv("ラベル.csv", encoding="utf-8-sig")
unit_keys = cm.UNIT_KEYS_AGG_SHA   # 販社合算で監視
common = dict(stable_start_m=36, baseline_len=12, monitor_end_m=90)

# (1) 単一設定で性能を見る
r = bt.evaluate(df, labels, unit_keys, **common,
                R=2.0, h=8.0,
                alpha_spike=0.001, min_count=3, burst_window=2)
print(r["power"], r["median_delay"], r["fa_per_unit_year"])

# (2) ドリフト側グリッドサーチ(ラベルは全タイプ渡し、power_driftで読む)
grid = bt.grid_search(df, labels, unit_keys, **common,
                      R_grid=[1.5, 2.0, 2.5, 3.0],
                      h_grid=[5, 8, 12, 18, 25],
                      alert_types=("drift",))

# (3) スパイク側グリッドサーチ
sgrid = bt.spike_grid_search(df, labels, unit_keys, **common,
                             alpha_grid=[0.01, 0.003, 0.001, 0.0003],
                             min_count_grid=[2, 3, 4],
                             burst_window=2)

# (4) 誤報予算と最低検知率を満たす操作点を早期順に
op = bt.suggest_operating_point(grid, max_fa_per_unit_year=0.05,
                                min_power=0.7, power_col="power_drift")
print(op.head())
```

注意: グリッドサーチには**全タイプのラベル**を渡すこと。一部タイプだけに
絞ると、他タイプの異常単位が「平常」扱いになり誤報が過大計上される。
関心タイプの検知率は `power_<タイプ>` 列で読む。

### 5-4. 操作点の選び方

`fa_per_unit_year` (誤報予算) と `power` (検知率) はトレードオフ。
組織の許容度に応じて「誤報を絞る」か「空振り覚悟で早める」かを決める。
予算を厳しくして候補が消える場合は、ベースライン窓や監視レンジの見直しも検討。

過剰適合対策: ラベル件数が少ない場合は時系列分割や leave-one-out で、
特定事例に合わせ込みすぎないようにする。

### 5-5. 決めた操作点の反映

確定した (R, h, alpha_spike, min_count, burst_window) を `run_monthly.py` の
`CONFIG` に書き込む。これで毎月の運用が固定の操作点で回る。

## 6. 安定化前モードのカーブ推定 (`earlylife_baseline.py`)

通常は `run_monthly.py` が内部で呼ぶため直接使う必要はないが、カーブの形を
確認したいときの使い方。

```python
import pandas as pd
from earlylife_baseline import estimate_earlylife_curve

leaders = df[df["開発コード"].isin(履歴の揃った先行機種)]
curve = estimate_earlylife_curve(
    leaders,
    group_keys=["事業コード", "SF-コード", "ランク"],
    max_keizoku=35,
)
# curve: 集団キー + 経過月 + lambda0_hat(平滑化済みカーブ) + lambda0_raw + n_models ...
```

推定の要点:
- 経験ベイズ Gamma-Poisson 部分プーリング。データの薄い機種を集団平均へ縮約する。
- ランク差は `group_keys` に「ランク」を含めて集団を分けることで物差しの歪みを防ぐ。
- 早期の期待値ゼロ連発を防ぐため、経過月方向の平滑化(`smooth_window`)と
  ゼロ下限(`rate_floor_frac`)を入れている。
- 推定カーブで新機種自身のスケールを合わせ直さないこと(異常をカーブが吸収するため)。

### 安定化前モードの限界(運用上の留意点)

- 先行機種が2〜3しかないとカーブの推定誤差が大きい。
- 発売直後は台数が小さくカウントも少ないため、本物でも信号が弱く、
  早期に見えるのは大きい異常に限られる。
- 位置づけは現場watchの置き換えではなく補助。優先度は安定期監視より低い。

## 7. Tableau連携

`run_monthly.py` が出力する縦長CSVを、そのままTableauのデータソースにする。

推奨ビュー:
- **実績 vs 期待**: `月次使用数` と `期待故障数` を経過月で重ねる。
- **CUSUM vs しきい値**: `CUSUM` と `しきい値h` を経過月で重ね、アラート点をハイライト。
- **worklist(要確認リスト)**: 各単位の最新月行を `月次p値` 昇順に並べる。
  `アラート種別`・`基準モード` で色分け/フィルタ。

すべて同一データソースで描けるよう、必要な属性列は出力CSVに含めてある。

## 8. プライバシー・運用境界

- 全ツールはネットワーク通信を行わない。データ・ラベル・検知結果はローカル完結。
- 社外共有が必要な場合も、共有するのは検知率などの集計値のみ。生データは出さない。
- 検証(backtest)も御社環境で完結する。ラベルの中身を外部が見る必要はない。

## 付録A: モジュール公開関数リファレンス

### cusum_monitor.py

- `monitor_all(df, unit_keys, stable_start_m, baseline_len, monitor_end_m, R, h, alpha_spike=None, min_count=2, burst_window=0, alpha_burst=None)`
  全監視単位に検知を適用し結果を縦結合。
- `monitor_unit(df_unit, stable_start_m, baseline_len, monitor_end_m, R, h, ..., lambda0_curve=None)`
  1単位の検知。`lambda0_curve` を渡すと安定化前モード。
- `aggregate_over_sha(df)` 販社合算(機種×部番)系列を作る。
- `poisson_cusum(usage, fleet, lambda0, R, h)` CUSUM本体。lambda0はスカラ/配列両対応。
- `spike_test(usage, fleet, lambda0, alpha_spike, ...)` スパイク/バースト検定。
- `estimate_lambda0(usage, fleet)` ベースライン窓から固定lambda0を推定。
- 定数: `UNIT_KEYS_AGG_SHA`(機種×部番) / `UNIT_KEYS_WITH_SHA`(機種×部番×販社)

### backtest.py

- `evaluate(...)` 単一パラメータ組での検知性能を返す。
- `grid_search(..., R_grid, h_grid)` ドリフト側 (R,h) の格子探索。
- `spike_grid_search(..., alpha_grid, min_count_grid)` スパイク側の格子探索。
- `suggest_operating_point(grid_result, max_fa_per_unit_year, min_power, power_col)`
  予算と検知率の制約を満たす操作点を早期順に返す。

### earlylife_baseline.py

- `estimate_earlylife_curve(df_leaders, group_keys, max_keizoku, ...)`
  先行機種から期待カーブ lambda0(t) を推定。
- `attach_curve_to_unit(df_unit, curve, group_keys)`
  監視対象単位の各月にカーブ値を引き当てて配列で返す。

## 付録B: パラメータ早見表

| パラメータ | 場所 | 意味 | 大きくすると |
|---|---|---|---|
| `R` | CONFIG/EARLYLIFE | 検知したい悪化倍率 | 大きい異常だけ狙う(感度↓) |
| `h` | CONFIG/EARLYLIFE | CUSUMしきい値 | 鳴りにくい(誤報↓・検知遅れ↑) |
| `alpha_spike` | CONFIG/EARLYLIFE | スパイク検定の有意水準 | 鳴りやすい(検知↑・誤報↑) |
| `min_count` | CONFIG/EARLYLIFE | スパイク最低カウント | 単発ノイズを弾く(誤報↓) |
| `burst_window` | CONFIG/EARLYLIFE | バースト合算月数(0で無効) | 数か月急増を拾う |
| `stable_start_m` | CONFIG | 安定期入口 | 監視開始を遅らせる |
| `baseline_len` | CONFIG | ベースライン窓長 | lambda0推定が安定(振り分け境界も後退) |
| `monitor_end_m` | CONFIG | 妥当性ウィンドウ末尾 | 末期まで見る(分母過大リスク) |
| `min_leaders` | EARLYLIFE | カーブ推定の最低先行機種数 | 安定化前監視が保守的に |
