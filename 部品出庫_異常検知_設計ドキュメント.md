# 部品出庫率による出庫状態の異常検知 — 設計ドキュメント

バージョン: 0.2（ドラフト）
目的: 設計方針の合意形成と、実装・チューニングの起点とすること

更新履歴:

- 0.1: 初版。

## 1. 背景と目的

修理部品の出庫状況から、製品（機種）の異常な故障増加を**早期に**検知したい。現場の経験則として「同じタイプの故障が一定割合を超えたらデータを確認した方がよい」という知見があり、これを定量的・自動的な監視に置き換えることを狙う。

### 1.1 検知したい異変の性質（重要）

本設計は、対象とする異変の性質を以下のように定める。これがすべての手法選択の前提になる。

- **主ターゲット**: 安定期に入った機種が、ある時点から劣化し始める「持続的な上振れ（ドリフト）」。設計起因・部品ロット起因などで、それまで平らだった故障ペースが少しずつ、あるいは段階的に上がっていくパターン。
- **副ターゲット**: 発売直後の初期不良の立ち上がり。ただしこちらは現場のwatchで捕捉できていることが多いため、自動検知の主眼ではない。

### 1.2 検知の対象外（明示的に捨てる）

- **突発スパイク（前月比急増）**: 製造を継続しているため、ある月から出庫数が突如跳ね上がるパターンは過去ほぼ存在しない。したがってスパイク検知系の手法は採用しない。捕まえたいのはスパイクではなくドリフトである。

## 2. データ概要

### 2.1 2つのデータマート

| | datamart_A | Datamart_B |
|---|---|---|
| 使用数系カラム | あり | あり |
| 販売台数系カラム（月次販売台数・累積販売台数・使用率） | **あり** | なし |
| 位置づけ | 販売台数が紐づき「率」を計算できるデータ | 使用数のみのデータ |

出庫率を扱う本件では **datamart_A が主データ**となる。

### 2.2 想定する粒度（一意キー）

```
事業コード × 開発コード × 部番 × 販社 × 年月
```

1行が「ある機種のある部品を、ある販社が、ある月にどれだけ使ったか」を表す縦長の時系列パネルデータ。

### 2.3 キーとなるカラムの役割

- `経過月`: 発売年月からの経過。製品ライフサイクル（初期高 → 安定 → 末期再上昇）の軸。期待値を作るうえでの基準時間。
- `累積販売台数`: 市場に出ている台数（保有台数）の近似。月次故障数を正規化する分母として使う。製造継続中のため毎月増加する。
- `SF-コード`: 開発コードを横断した類似部品のグルーピング。**新製品の期待値を先行機種から借用する場面でのみ**使用する（後述）。
- `月次使用数`: 監視対象の生の信号。低頻度（多くの部番で月0〜数個）。

## 3. 設計判断の経緯（なぜこの設計に至ったか）

実装者が後から「なぜこうなっているのか」を理解できるよう、検討の分岐と却下理由を残す。

### 3.1 スパイク検知 → 却下

突発増加が過去ほぼ存在しないため、前月比の急増を見る手法は信号源が無い。ドリフト検知に的を絞る。

### 3.2 累積使用率の固定しきい値 1本 → 不採用（単独では）

`使用率`（累積使用数 ÷ 累積販売台数）に共通しきい値を当てる案。シンプルだが2つの欠点がある。
- **遅行指標**: 累積がしきい値を超える頃には問題は数か月前から進行済み。早期検知の目的と相反する。
- **経過月との交絡**: 累積率は単調増加するため、古い機種ほど中身が正常でも率が高く出る。固定しきい値では古い機種ばかり引っかかる。

経験則の表現としては残せるが、早期検知のエンジンには使わない。

### 3.3 SF-コードで束ねて監視 → 却下

複数機種を1系列に合算すると「どの機種で異変が起きたか」が消える。機種単位で監視したいという目的と逆行するため却下。

### 3.4 SF-コードで他機種と横並び比較（集団を物差しにする）→ 却下

「同種他機種の正常分布から外れたら異常」とする案。2つの理由で成立しない。
- **機種ランクの交絡**: 高ランク（高価格）機種は元々修理率が高い。ランク混在の集団を物差しにすると誤判定する。ランクで揃えると次の問題が悪化する。
- **機種数の不足**: SF-コードあたり 2〜7機種（少ないと2〜3）。集団分布を推定するには少なすぎ、さらにランクで割ると1〜2機種しか残らない。統計的な基準にならない。

### 3.5 自己基準のドリフト検知 → **採用**

各機種を「同種他機種」ではなく「**その機種自身の過去（平常運転）**」と比較する。
- ランク問題が自動的に消える（自分自身が基準なので機種間水準差は無関係）。
- 機種数の少なさが問題でなくなる（比較対象は自分の時系列）。
- 「いつものペースから逸脱し始めた瞬間」を狙うため、早期検知の目的に直結する。

## 4. 監視指標の定義

### 4.1 1台あたり月次故障ペース

生の `月次使用数` は保有台数増加に伴い自然に増える。これに惑わされないため、保有台数で正規化した月次レートを基本指標とする。

```
月次レート_t = 月次使用数_t / 累積販売台数_t
```

- 累積使用率（遅行）ではなく月次側を見ることで、ドリフトに早く反応する。
- 注意（仮定）: `累積販売台数` は廃棄・退役を考慮しない保有台数の近似。厳密な稼働台数ではない点は承知のうえで分母として用いる。詳細は 4.3 を参照。

### 4.3 妥当性ウィンドウ（分母＝稼働台数近似の限界への対処）

`累積販売台数` は実稼働台数の近似だが、年数が経つと退役で実稼働が減り、入庫も減る。一方で累積販売台数は製造継続のため増え続ける。この乖離は**上側CUSUMでは誤報を増やす方向ではなく、本物の上昇を見えにくくする（マスクする）方向**に効く。期待故障数（= λ0 × 累積販売台数）が過大になり、実績との差が下にずれるためである。

対処は、近似が成立する経過月レンジ（妥当性ウィンドウ）に監視を限定すること。監視終了の経過月 `monitor_end_m` をこのウィンドウの末尾とする。

- 主ターゲットの安定期は退役が顕著になる前の区間なので、妥当性ウィンドウは安定期の範囲とおおむね一致させる。
- データから決めるなら、月次販売台数（累積の増分）が十分立っている、あるいは累積使用数がまだ伸びている区間を採る。増分がほぼ0かつ累積使用数も頭打ちした以降は対象外。
- 退役の速さが事業コード／部品種で異なれば、`monitor_end_m` を分けて設定する。

この限界ゆえ、本設計は「末期の劣化」までは守備範囲としない。末期まで見たい場合は、累積販売台数ではなく退役を織り込んだ稼働台数推定（生存曲線等）が別途必要になる。

### 4.2 安定期の切り出し

主ターゲットは安定期の劣化。`経過月` を用いて、ライフサイクルが平らになる区間を安定期として定義する。

- 安定期の開始・終了の経過月は、事業コードや部品種ごとに異なる可能性があるため、初期は経験的に決め、データで見直す（→ 9章 未決事項）。

## 5. 検知エンジン

### 5.1 基本思想：固定ベースラインからの片側ドリフト検知

- 平常水準（期待値とばらつき）を**安定期の初期で固定**する。移動平均などで期待値を更新し続けると、じわじわ進む劣化を期待値が追随吸収してしまい検知できなくなる。「何を平常とみなすか」を固定するのがこの種の検知の肝。
- 上振れ（故障増加）のみが関心事なので、**片側（上側）**の監視とする。

### 5.2 推奨手法：ポアソンCUSUM（低頻度カウント向け）

月次使用数は低頻度の離散カウントで、保有台数も毎月増える。これに最も素直に対応できるのがポアソンCUSUMである。月次レートを正規分布として扱う前に、カウントそのものをポアソン過程として監視する。

考え方:
- 平常時の真のレートを `λ0`（1台・1か月あたりの故障率）とする。安定期初期から推定し固定。
- ある月の期待故障数は保有台数に比例して増える: `μ0_t = λ0 × 累積販売台数_t`。これにより**保有台数の増加は自動的に期待値側に吸収され、レートの上昇分だけがアラート対象**になる。
- 検知したい上昇後レートを `λ1`（> λ0）とする。「どの程度の悪化を捕まえたいか」で決める。
- 片側上側CUSUM:

```
参照値:    k_t = (λ1 - λ0) × 累積販売台数_t / ln(λ1 / λ0)
統計量:    S_t = max(0, S_{t-1} + (月次使用数_t - k_t))
アラート:  S_t > h
```

`h` がしきい値（感度）。`S_t` は普段0付近にとどまり、レートが λ0 から λ1 方向へ持続的に上がると積み上がっていく。単月では判断できない小さな上振れも、累積で前倒しに捕まえられる。

### 5.3 代替手法：ガウスCUSUM / EWMA（参考）

月次レートを連続量として扱う場合の標準形。実装が直感的で、ポアソンの前提が崩れる場合の代替になる。

```
S_t = max(0, S_{t-1} + (月次レート_t - μ0 - k))     # k は σ 単位で δ/2 程度
アラート: S_t > h                                    # h は 4〜5σ 程度が目安
```

低頻度ゆえ月次レートのばらつきが大きいので、第一選択はポアソンCUSUM、これは補助と位置づける。

## 6. 監視単位 — 二段構え

低頻度データを販社まで割るとさらに細切れになる。一方で「特定販社だけ上振れ」は拾いたい情報。そこで2階層で持つ。

1. **機種 × 部番（販社合算）**: 感度を最優先。全体の劣化をいち早く捕まえる主監視。
2. **機種 × 部番 × 販社**: 偏りの所在を切り分ける副監視。合算側でアラートが出た後、どの販社起因かを特定する。

運用上は合算側で気づき、販社側でドリルダウンする流れを想定。

## 7. 新製品の扱い（副ターゲット・ハイブリッド）

発売直後は自分の平常運転カーブが引けない。この期間のみ、**同じSF-コードを持つ先行機種が描いたライフサイクルカーブの「形」を暫定期待値として借用**する。

- 点推定の物差し（4章で却下したもの）としてではなく、「カーブ形状の参考」としてなら 2〜3機種でも利用できる。
- 自分の履歴が十分たまった時点で、自己基準（5章）に切り替える。

優先度は低い（初期はwatchで補えているため）。主エンジンの実装後に追加する。

## 8. Tableau連携のためのデータ構造

監視結果を Tableau で可視化・共有する前提で、出力テーブルは以下の縦長構造を推奨する。1行 = 1監視単位 × 1年月。

| カラム | 内容 |
|---|---|
| 事業コード / 開発コード / 部番 / 販社 | 監視単位キー（販社は合算時はNULLまたは "ALL"） |
| 製品名 / ジャンル / カテゴリー1-3 / 部品名 / SF-コード | フィルタ・ドリルダウン用の属性 |
| 年月 / 経過月 | 時間軸 |
| 月次使用数 | 実績（生カウント） |
| 累積販売台数 | 保有台数近似 |
| 期待故障数 `μ0_t` | λ0 × 累積販売台数。実績との対比でプロット |
| CUSUM統計量 `S_t` | 累積和。閾値超えで折れ線が立ち上がる |
| しきい値 `h` | 参照線 |
| アラートフラグ | S_t > h の月を1。Tableau で色・形で強調 |
| ライフサイクル区分 | 立ち上がり / 安定期 / 末期 など |

これにより Tableau 側では、(a) 実績 vs 期待故障数の対比、(b) CUSUM統計量 vs しきい値、(c) アラート点のハイライト、の3点を同一データソースで描ける。

## 9. パラメータと未決事項

実装前に決める必要がある項目。

### 9.1 感度パラメータ（k / h）のトレードオフ

- 感度を上げる（h を下げる / λ1 を λ0 に近づける）→ 早く鳴るが誤報増。
- 感度を下げる → 確実だが検知が遅れる。
- 低頻度ゆえ、本当に小さな上振れの確定には数か月の蓄積が要る。「2〜3か月かかっても誤報が少ない方がよいか / 空振り覚悟で1か月でも早く知りたいか」の運用方針で値を当てる。
- **チューニングは過去のアラート相当事例（ラベル）に対するバックテストで行う**。ラベルは社外に出せないため、付属の検証ツール群（`cusum_monitor.py` / `backtest.py`）を御社環境で実行し、検知率・検知遅れ・誤報率を見ながら `(R, h)` を決める。手順は検証手順書を参照。データやラベルの中身を共有する必要はなく、共有するとしても集計値（検知率等）のみ。
- ラベル件数が少ない場合の過剰適合対策（時系列分割・leave-one-out）は検証手順書5章を参照。

### 9.2 その他

- 安定期の経過月レンジの定義（事業コード／部品種別ごとに異なるか）。
- λ0 推定に使う安定期初期の窓幅（何か月分を平常とみなすか）。`baseline_len`。
- 妥当性ウィンドウ `monitor_end_m` の設定（4.3）。事業コード／部品種で分けるか。
- 月次0の行が欠落していないか（CUSUMは0月も「上振れなし」の証拠として使うため、0で補完しておく）。

### 9.3 決定済み事項

- **分母（累積販売台数）の稼働台数近似問題**: 妥当性ウィンドウで監視レンジを限定して対処（4.3）。末期は守備範囲外とする。
- **B限定部品（販売台数が取れず率が出せないもの）**: 監視対象に含めない。

## 10. 実装ステップ（提案）

1. datamart_A から監視単位ごとの時系列を構築（4.1 の月次レート、安定期切り出し）。
2. 安定期初期から λ0 を推定・固定。
3. ポアソンCUSUM を実装し、機種×部番（販社合算）で監視。
4. 過去データでバックテストし、k/h を調整（9.1）。
5. 販社別の副監視を追加（6章）。
6. Tableau 出力テーブルを生成（8章）。
7. 新製品向けSF-コード借用ロジックを追加（7章）。

## 11. 付属の検証ツール群

実装・検証を御社環境で完結させるため、以下を同梱する（詳細は検証手順書）。

- `cusum_monitor.py`: 時変平均ポアソンCUSUMの監視エンジン。ベースライン推定（λ0）、妥当性ウィンドウでの監視、販社合算／販社別の切り替えを提供。
- `backtest.py`: ラベルに対する検知性能の計測（検知率・検知遅れ・誤報率）と `(R, h)` のグリッドサーチ、操作点の提案。
- `synthetic_demo.py`: 疑似データで仕組みの動作を確認するデモ兼使用例。
- 検証手順書: 入力フォーマット、実行手順、指標の読み方、過剰適合対策、Tableau連携、プライバシー。

これらはネットワーク通信を行わず、データ・ラベル・結果はローカルに留まる。

## 付録: 用語

- **ドリフト**: 突発的でなく、持続的・漸進的に水準がずれていく変化。本件の主検知対象。
- **CUSUM（累積和管理図）**: 観測値と期待値の差を累積し、小さな変化が持続していることを早期に検知する管理図手法。
- **片側監視**: 上振れ（または下振れ）の一方向のみを対象とする監視。本件は故障増加＝上側のみ。
